## 1. 작업 배경

쿠팡 판매/광고/프로모션 데이터는 원본상 `optionId`, `optionName`, `adExecutionOptionId`, `conversionOptionId` 같은 옵션 단위 식별자와 텍스트를 포함한다. 하지만 실제 운영에서는 같은 상품의 같은 옵션이라도 옵션 ID가 자주 바뀌거나 여러 개로 나뉘는 경우가 있어 `optionId`를 안정적인 그룹 기준으로 쓰기 어렵다.

현재 앱은 `CoupangProduct`를 내부 기준 상품으로 사용한다. 판매행, 광고행, 프로모션행은 업로드 시 매칭 규칙을 통해 `coupangProductId`, `spendProductId`, `conversionProductId` 등에 연결되고, 수익 계산은 이 내부 상품 기준으로 수행된다. 다만 현 운영상 `CoupangProduct`가 옵션/구성 단위처럼 쓰이고 있어, 예를 들어 아래와 같은 상품을 하나의 통합 상품으로 보기 어렵다.

```txt
제품그룹: 자이로볼
- 옵션/구성 상품: 자이로볼 메딕
- 옵션/구성 상품: 자이로볼 챌린지
```

요구사항은 각 옵션/구성 상품의 원가, 수수료, 배송비, 반품비, 광고비, 마진 등을 기존처럼 정확히 계산한 뒤, 그 계산 결과를 한 번 더 합산해서 하나의 제품그룹 성과로 볼 수 있게 하는 것이다.

## 2. 최종 방향

권장 구조는 DB에 그룹 식별 축을 추가하고, 실제 숫자 집계는 API에서 기존 옵션별 계산 결과를 재집계하는 방식이다.

핵심 원칙:

1. 원본 데이터 행은 계속 옵션/구성 단위로 보존한다.
2. `optionId`는 그룹 기준으로 쓰지 않는다.
3. 기존 `CoupangProduct`는 옵션/구성 단위 상품으로 유지한다.
4. 새 `CoupangProductGroup`을 추가해 여러 `CoupangProduct`를 하나의 통합 제품그룹으로 묶는다.
5. 수익 계산은 먼저 기존 `CoupangProduct`별로 수행하고, 같은 `groupId`를 가진 결과 row들을 합산해 그룹 row를 만든다.
6. 매출, 원가총액, 수수료총액, 배송비총액, 반품비총액, 광고비, 마진액 등 금액/수량 값은 합산한다.
7. 마진율, ROAS 같은 비율은 옵션별 비율을 평균내지 말고 그룹 합산값 기준으로 다시 계산한다.
8. 그룹이 없는 상품은 `groupBy=group`에서도 임의의 "미분류" 하나로 묶지 말고, 자기 자신을 하나의 독립 row처럼 유지한다. 서로 관련 없는 상품이 합쳐지는 것을 막기 위함이다.

## 4. 데이터 모델 설계

### 추가 모델

`CoupangProductGroup` 모델을 추가한다.

권장 Prisma 모델:

```prisma
model CoupangProductGroup {
  id           String           @id @default(uuid()) @db.Uuid
  standardName String           @unique @map("standard_name")
  displayName  String           @map("display_name")
  sortOrder    Int              @default(100) @map("sort_order")
  isActive     Boolean          @default(true) @map("is_active")
  createdAt    DateTime         @default(now()) @map("created_at")
  updatedAt    DateTime         @updatedAt @map("updated_at")
  products     CoupangProduct[]

  @@index([isActive, sortOrder])
  @@map("coupang_product_groups")
}
```

`CoupangProduct`에는 nullable FK를 추가한다.

```prisma
model CoupangProduct {
  id           String               @id @default(uuid()) @db.Uuid
  groupId      String?              @map("group_id") @db.Uuid
  standardName String               @unique @map("standard_name")
  displayName  String               @map("display_name")
  sortOrder    Int                  @default(100) @map("sort_order")
  isActive     Boolean              @default(true) @map("is_active")
  createdAt    DateTime             @default(now()) @map("created_at")
  updatedAt    DateTime             @updatedAt @map("updated_at")
  group        CoupangProductGroup? @relation(fields: [groupId], references: [id], onDelete: SetNull)
  productRules CoupangProductRule[]
  costRules    CoupangCostRule[]
  saleLines    CoupangSaleLine[]
  promotionPrices CoupangPromotionPrice[]
  spendAdMetrics CoupangAdMetric[]  @relation("CoupangAdSpendProduct")
  conversionAdMetrics CoupangAdMetric[] @relation("CoupangAdConversionProduct")

  @@index([isActive, sortOrder])
  @@index([groupId])
  @@map("coupang_products")
}
```

주의:

- `groupId`는 nullable로 둔다.
- 그룹 삭제는 soft delete(`isActive=false`)를 기본으로 한다.
- 그룹 hard delete가 필요한 경우에도 연결 상품은 `onDelete: SetNull`로 남긴다.
- 처음부터 그룹 이력 테이블을 만들지 않는다. 나중에 기간별 그룹 소속 이력이 필요해지면 `CoupangProductGroupMembership(productId, groupId, validFrom, validTo)`로 확장한다.

권장 타입:

```ts
type CoupangGroupBy = "product" | "group";

function parseCoupangGroupBy(value: unknown): CoupangGroupBy {
  return value === "group" ? "group" : "product";
}
```

## 6. 백엔드 구현 세부 계획

### ProductProfitRow 타입 확장

현재 backend `ProductProfitRow`에는 이미 `productId`가 있다. 그룹 표시를 위해 optional 필드를 추가한다.

```ts
type ProductProfitRow = {
  rowType?: "PRODUCT" | "GROUP";
  productId: string;
  productName: string;
  groupId?: string | null;
  groupName?: string | null;
  childProductCount?: number;
  children?: ProductProfitRow[];
  ...
};
```

주의:

- 기존 프론트가 `productName`만 쓰고 있으므로, group row에서도 `productName`에 그룹명을 넣으면 최소 변경으로 화면 표시가 가능하다.
- `children`은 선택사항이다. 처음에는 API 응답에 포함하되 프론트에서는 당장 펼침 UI를 만들지 않아도 된다. 다만 다음 작업자가 확장하기 쉽다.
- 순환 구조를 피하려면 group row의 `children`에는 children 자신의 `children`을 넣지 않는다.

### `buildProductProfitRows()`는 기존 계산 단위 유지

`buildProductProfitRows(range)`는 계속 `CoupangProduct` 단위 row를 만든다. 여기에 그룹 로직을 직접 섞기보다, 아래처럼 래핑한다.

```ts
async productProfit(query: { from?: string; to?: string; groupBy?: string }) {
  const range = parseDateRange(query.from, query.to);
  const productRows = await this.buildProductProfitRows(range);
  const rows = parseCoupangGroupBy(query.groupBy) === "group"
    ? await this.groupProductProfitRows(productRows)
    : productRows;
  return { period: { from: range.from, to: range.to }, groupBy: parseCoupangGroupBy(query.groupBy), rows };
}
```

### 그룹 집계 helper 추가

권장 helper 이름:

```ts
private async groupProductProfitRows(rows: ProductProfitRow[]): Promise<ProductProfitRow[]>
```

동작:

1. rows에서 `productId` 목록을 뽑는다.
2. `this.prisma.coupangProduct.findMany({ where: { id: { in: productIds } }, include: { group: true } })`로 그룹 정보를 조회한다.
3. 각 row에 대해 group key를 만든다.
   - group이 있으면 `group:${group.id}`
   - group이 없으면 `product:${row.productId}`
4. 같은 group key끼리 합산한다.
5. group이 없는 상품은 이름/ID를 원래 product row 기준으로 둔다.
6. group이 있는 경우 group row의 `productName`은 `group.displayName`으로 둔다.

권장 집계 규칙:

합산 필드:

- `matchedSalesLineCount`
- `salesQuantity`
- `orderCount`
- `salesKrw`
- `cancelAmountKrw`
- `netSalesKrw`
- `productCostKrw`
- `salesFeeKrw`
- `shippingCostKrw`
- `returnCostKrw`
- `extraCostKrw`
- `adSpendKrw`
- `adConversionSalesKrw`
- `adConversionQuantity`
- `organicSalesKrw`
- `totalCostKrw`
- `marginKrw`

재계산 필드:

- `marginRate = safeDivide(group.marginKrw, group.netSalesKrw)`
- `roas = safeDivide(group.adConversionSalesKrw, group.adSpendKrw)`

표시상 주의 필드:

- `salePriceKrw`, `baseSalePriceKrw`, `promotionPriceKrw`는 옵션별로 다를 수 있다.
- 그룹 row에서 단일 판매가를 강제로 표시하지 않는 것이 안전하다.
- 권장: 그룹 row에서는 이 값들을 `null`로 두고 `priceSource = "MIXED"`를 추가하거나, 기존 union 타입 확장이 부담되면 `priceWarnings`에 `GROUP_MIXED_PRICE`를 넣는다.
- `saleMethod`도 여러 값이면 `null` 또는 `"MIXED"`가 안전하다. 현재 타입은 `string | null`이므로 `"MIXED"` 문자열 사용 가능.

nullable 금액 합산 규칙:

- 모든 child 값이 `null`이면 group 값도 `null`.
- 하나라도 숫자가 있으면 숫자만 합산한다.
- 예: 어떤 옵션은 원가 규칙 누락으로 `totalCostKrw=null`, 다른 옵션은 정상 계산이면 그룹 `totalCostKrw`를 숫자로 두되 warning에 `GROUP_HAS_MISSING_COST_RULE`을 넣는다.

권장 helper:

```ts
function sumNullable(values: Array<number | null | undefined>) {
  const numbers = values.filter((value): value is number => Number.isFinite(value));
  return numbers.length > 0 ? numbers.reduce((sum, value) => sum + value, 0) : null;
}
```

`ruleStatus` 집계 규칙:

- 모든 child가 `OK`면 `OK`
- 하나라도 `MISSING_COST_RULE`이면 `MISSING_COST_RULE`
- 하나라도 `UNMATCHED`이면 `UNMATCHED`
- 부분 누락을 더 명확히 보여주고 싶다면 warnings에 `GROUP_HAS_MISSING_COST_RULE` 추가

`warnings` 집계 규칙:

- child warnings를 중복 제거해 합친다.
- 그룹 자체 warning을 추가한다.
  - 가격이 서로 다르면 `GROUP_MIXED_PRICE`
  - 판매방식이 서로 다르면 `GROUP_MIXED_SALE_METHOD`
  - 일부 child 원가 누락이면 `GROUP_HAS_MISSING_COST_RULE`

정렬:

- 기존과 동일하게 `netSalesKrw desc`, `productName asc`.

### dashboard 변경

현재 `dashboard()`는 `productProfit(query)`를 호출하고 그 rows로 summary를 만든다. `groupBy`를 전달하면 된다.

주의:

- summary 총액은 product view와 group view가 같아야 한다.
- group view로 바꿔도 top rows만 달라져야 한다.
- `missingCostRuleCount`는 group view에서 그룹 row 기준으로 카운트하면 product view와 숫자가 달라질 수 있다. 이것이 의도인지 결정해야 한다.
  - 권장: dashboard summary는 실제 문제 상품 수를 보여주는 것이 더 유용하므로 productRows 기준으로 계산하고, rows만 group view로 보여준다.
  - 구현이 단순해야 한다면 group rows 기준으로 유지해도 되지만, 계획서상 권장은 productRows 기준이다.

권장 구현:

```ts
async dashboard(query) {
  const range = parseDateRange(query.from, query.to);
  const productRows = await this.buildProductProfitRows(range);
  const rows = parseCoupangGroupBy(query.groupBy) === "group"
    ? await this.groupProductProfitRows(productRows)
    : productRows;
  const totals = productRows.reduce(...); // summary는 productRows 기준
  return { period, groupBy, summary, rows: rows.slice(0, 20) };
}
```

### adsAnalysis 변경

현재 `adsAnalysis()`는 `spendProductId + campaignName + adGroupName` 기준으로 광고 데이터를 묶는다. group view에서는 `spendProduct.groupId` 기준으로 묶어야 한다.

변경 방향:

2. groupBy가 `product`이면 기존 key 유지.
3. groupBy가 `group`이면 key를 `[resolvedGroupKey, campaignName, adGroupName]`로 만든다.
4. group이 있는 경우 productName은 group displayName으로 표시한다.
5. group이 없는 경우 기존 상품명으로 유지한다.

주의:

- 전환 상품 그룹 기준 분석까지 별도로 필요하면 추가 설계가 필요하지만, 이번 범위에서는 현재 adsAnalysis의 의미인 "광고 집행 상품 기준"을 group으로 올리는 것으로 충분하다.

## 11. 주의사항

### optionId는 기준으로 쓰지 말 것

이번 요구의 출발점은 `optionId`가 불안정하다는 점이다. 새 그룹 로직에서 `optionId`를 그룹 식별자로 쓰면 같은 문제가 반복된다. `optionId`는 원본 추적/디버깅 용도로만 유지한다.

### 원본 행에 groupId를 저장하지 말 것

초기 구현에서는 `CoupangSaleLine`, `CoupangAdMetric`, `CoupangPromotionPrice`에 groupId를 추가하지 않는다. 중복 저장이 되고, 그룹 변경 시 과거 행을 모두 갱신해야 한다. 지금은 `CoupangProduct.groupId` 하나로 충분하다.

### 매핑 규칙은 그룹이 아니라 상품에 연결

매핑 규칙은 계속 `CoupangProductRule -> CoupangProduct` 구조를 유지한다. 그룹에 직접 매핑하면 옵션별 원가/판매가/프로모션 차이를 잃는다.

### 그룹별 판매가 표시

그룹 row에서 판매가/프로모션가는 단일 값으로 표현하기 어렵다. 처음 구현에서는 `null` 또는 `MIXED` 경고로 처리한다. 가중평균 판매가가 필요하면 별도 필드로 `averageSalePriceKrw = netSalesKrw / salesQuantity`를 추가하는 식이 낫다.

### 일부 원가 누락

그룹 안의 일부 옵션만 원가 규칙이 없을 수 있다. 이 경우 group row의 총비용/마진을 숫자로 보여줄지 null로 둘지 결정이 필요하다.

권장:

- 숫자가 있는 child는 합산한다.
- 누락 child가 있으면 warning을 명확히 표시한다.
- 사용자가 "이 그룹 수익은 일부 원가 누락으로 불완전하다"는 사실을 알 수 있어야 한다.

### 기존 API 호환성

`groupBy` 기본값은 `product`여야 한다. 기존 화면이 query를 바꾸기 전에도 응답 형태가 크게 달라지면 안 된다.

### 정렬과 그룹 없는 상품

그룹 없는 상품들을 하나의 "미분류"로 묶으면 안 된다. 관련 없는 상품들이 합쳐져 수익표가 오염된다. group view에서도 group 없는 상품은 각각 독립 row로 유지한다.

### 인코딩

일부 프론트 파일의 한글이 터미널 출력에서 깨져 보일 수 있다. 기존 파일 전체를 재인코딩하거나 unrelated text를 고치지 않는다. 이번 작업 범위는 그룹 기능에 한정한다.

### 테스트 DB 상태

로컬 DB에 기존 상품 데이터가 있다면 migration 후 groupId는 모두 null이다. UI에서 그룹을 생성하고 상품을 연결해야 group view가 의미 있게 합쳐진다.

3. 쿠팡 상품 설정 화면에서 그룹을 생성하고 상품에 연결할 수 있다.
4. 기존 옵션/구성 상품별 수익표는 유지된다.
5. 수익표에서 제품그룹별 보기로 전환하면 같은 그룹의 상품들이 하나의 row로 합쳐진다.
6. 그룹 row의 마진율/ROAS는 합산값 기준으로 재계산된다.
7. 대시보드와 일일 리포트에서도 옵션별/제품그룹별 전환이 가능하다.
8. 광고 분석에서도 groupBy가 동작한다.
9. `groupBy`를 주지 않은 기존 API 호출은 기존과 같은 의미로 동작한다.
10. 백엔드 테스트가 통과한다.

## 14. 구현 시 참고할 기존 재사용 포인트

재사용할 것:

- `standardProductName()` helper: 그룹 standardName 정규화에 재사용.
- `requiredString()`, `optionalString()`, `optionalNullableString()`, `numberOrDefault()`: 그룹 CRUD body parsing에 재사용.
- `safeDivide()`: 그룹 marginRate, roas 재계산에 재사용.
- `groupBy<T>()`: 이미 service 하단에 있으므로 그룹 관련 map 구성에 재사용 가능.
- `calculateCoupangProfit()`: 절대 그룹용으로 다시 호출하지 말고, 옵션별 계산에만 계속 사용.
- `buildProductProfitRows()`: 기존 옵션별 계산의 중심. 여기를 최대한 보존하고 결과 재집계 helper를 추가.
- `DataTable`, `KpiCard`, `apiGet/apiPost/apiPatch/apiDelete`: 프론트 기존 패턴 유지.

새로 만들 것:

- `CoupangProductGroup` 모델
- group CRUD API
- product setting의 groupId 연결
- product profit row 그룹 집계 helper
- groupBy query parsing
- 화면별 groupBy 전환 UI

피할 것:

- `optionId` 기반 그룹핑
- 원본 행에 groupId 중복 저장
- 그룹 row에서 원가/마진을 다시 단가 기준으로 계산
- 옵션별 비율을 평균내는 방식
- 매핑 규칙을 그룹에 직접 붙이는 방식
- 기존 한글/레이아웃을 광범위하게 정리하는 부수 작업
