# 0610 수정계획 2: Change Logs 제품 단위 재구성

## 1. 이번 요청의 핵심 이해

`Change Logs` 탭은 새 화면으로 갈아엎는 것이 아니라, 현재 실행 중인 프로그램의 화면 구조를 최대한 유지한 상태에서 내용만 제품 단위로 바꾼다.

유지할 구조:

- 왼쪽 사이드바
- 상단 날짜 선택 바
- `Change Logs` 페이지 제목 영역
- 좌측 목록 패널
- 우측 상세 패널
- 우측 상세 패널 안에서 섹션이 아래로 이어지는 기존 박스형 구조

바꿀 내용:

- 좌측 광고 소재 박스를 제품명 박스로 변경
- 좌측 제품 박스에는 `제품명`, `광고비`만 표시
- 제품 박스는 기존 광고소재 박스보다 작게 만들고, 좌측 패널 안에서 한 줄에 가능한 만큼 여러 개 들어가게 배치
- 우측 상세 패널은 기존 박스 구조를 유지하되, 선택 제품 기준으로 정보 표시
- 활성화 광고와 비활성화 광고는 같은 줄에 나란히 배치
- 새 기록 입력 영역은 기존 기록 목록보다 위에 배치
- 기록 목록은 고정 높이 박스 안에 넣고, 일정 세로 길이를 넘으면 내부 스크롤
- 기록 분류는 `NOTE`, `KEEP`, `WATCH` 등으로 나누지 않고 단일 `기록`으로만 관리
- 기록 입력은 `기록일`, `기록` 텍스트박스 하나만 유지
- `다음 확인일`, `사유`, `메모` 입력은 삭제
- 날짜 기본값은 윈도우 로컬 날짜 기준 `오늘`

## 2. 현재 구조 기준 변경 방향

### 2.1 프론트엔드 대상

주요 대상 파일:

- `apps/web/src/app/change-logs/page.tsx`
- `apps/web/src/app/globals.css`

현재 구조는 대략 다음과 같다.

```text
Change Logs

[좌측 광고소재 목록 패널] [우측 광고소재 상세 패널]
```

변경 후 구조는 다음과 같아야 한다.

```text
Change Logs

[좌측 제품 목록 패널] [우측 선택 제품 상세 패널]
```

중요한 점은 좌우 패널 구조를 유지하는 것이다. 제품 목록을 페이지 전체 상단 그리드로 빼거나, 새로운 대시보드형 화면으로 만들지 않는다.

### 2.2 백엔드 대상

주요 대상 파일:

- `apps/api/src/change-logs/change-logs.controller.ts`
- `apps/api/src/change-logs/change-logs.service.ts`
- `apps/api/prisma/schema.prisma`

현재 광고 소재 중심 API:

- `GET /change-logs/creatives`
- `GET /change-logs/creatives/:creativeId`
- `POST /change-logs/creatives/:creativeId/logs`

제품 중심 API로 추가 또는 교체할 엔드포인트:

- `GET /change-logs/products?date=YYYY-MM-DD`
- `GET /change-logs/products/:productId?date=YYYY-MM-DD`
- `POST /change-logs/products/:productId/logs`

기존 creative API는 바로 제거하지 말고, 제품 단위 화면이 안정화된 뒤 정리한다.

## 3. 좌측 제품 목록 패널

현재 좌측 패널의 역할은 그대로 유지한다.

기존:

- 광고 소재 목록
- 각 박스에 소재명, 배치 수, 광고비, 구매수, 클릭수, 최근 로그 등 여러 정보 표시

변경:

- 제품 목록
- 각 박스에는 딱 `제품명`, `광고비`만 표시
- 구매수, 클릭수, 소재 수, 최근 로그 문구는 표시하지 않음

제품 박스 형태:

```text
[두줄바]
[$348.20]
```

배치 기준:

- 좌측 패널 안에서 `repeat(auto-fit, minmax(138px, 1fr))` 같은 조밀한 그리드 사용
- 한 줄에 2개 이상 들어갈 수 있으면 들어가게 구성
- 박스 높이는 기존 광고소재 박스보다 낮게 유지
- 선택된 제품은 기존처럼 배경색 또는 테두리로 active 표시

검색:

- 기존 검색 박스 위치 유지
- placeholder는 `제품명 검색`

## 4. 우측 선택 제품 상세 패널

우측 패널은 기존 상세 패널의 박스 구조를 유지한다.

상단:

- 제품명
- 선택 날짜 기준 설명
- `제품 기록` 또는 유사한 작은 badge

요약 박스:

- 광고비
- 활성화 광고 수
- 비활성화 광고 수

요약 박스는 기존 metric box 스타일을 재사용한다.

## 5. 활성화/비활성화 광고 영역

활성화 광고와 비활성화 광고는 같은 줄에 배치한다.

이전 목업처럼 `활성화 광고` 섹션 하나가 한 줄 전체를 차지하고, 그 아래에 `비활성화 광고` 섹션이 다시 한 줄 전체를 차지하면 안 된다. 표시하는 정보가 `광고소재명`, `광고비`, `ROAS`뿐이라 세로 공간을 크게 쓰는 구조가 과하다.

권장 배치:

```text
[활성화 광고 박스] [비활성화 광고 박스]
```

각 박스 내부 표 컬럼:

- 광고소재명
- 광고비
- ROAS

예시:

```text
활성화 광고                         비활성화 광고
광고소재명      광고비    ROAS       광고소재명      광고비    ROAS
소재01          $120.50   2.84       소재07          $12.00    0.74
소재03          $88.20    1.92       소재09          $4.30     -
```

CSS 방향:

```css
.ad-status-grid {
  display: grid;
  grid-template-columns: repeat(2, minmax(0, 1fr));
  gap: 8px;
}

.status-ad-panel {
  min-width: 0;
  border: 1px solid var(--line);
  border-radius: 8px;
  background: #fbfcfd;
  padding: 10px;
}
```

모바일 또는 좁은 화면에서는 1열로 내려가도 된다.

## 6. 새 기록과 기록 목록 순서

우측 상세 패널 하단 순서는 다음이어야 한다.

```text
새 기록
기록
```

즉, 사용자가 기록을 입력하는 `새 기록` 폼이 기존 기록 목록보다 위에 있어야 한다. 현재 기록을 먼저 읽고 맨 아래까지 내려가서 입력하는 구조가 아니라, 제품 상세를 본 뒤 바로 기록을 남길 수 있게 한다.

새 기록 폼:

- `기록일` date input
- `기록` textarea
- `저장` 버튼

삭제할 항목:

- `NOTE`, `KEEP`, `WATCH` 등 기록 유형 선택 UI
- `다음 확인일`
- `사유`
- `메모`
- 관련 광고세트 체크박스

## 7. 기록 목록 스크롤 박스

기록 목록은 별도 박스로 감싼다.

요구사항:

- 기록 목록 영역에 고정 또는 최대 세로 길이 설정
- 기록이 일정 길이 이상 많아지면 페이지 전체가 길어지는 대신 기록 박스 내부에서 스크롤
- 각 기록은 `기록일`, `기록 텍스트`만 표시
- 기록 분류 badge는 표시하지 않음

권장 CSS:

```css
.log-scroll-box {
  max-height: 220px;
  overflow-y: auto;
  border: 1px solid var(--line);
  border-radius: 8px;
  background: #fff;
  padding: 8px;
}
```

기록 표시 예시:

```text
기록
[스크롤 박스]
  2026-06-09
  블랙 2팩 소재 광고비 증액 후 ROAS 유지.

  2026-06-08
  신규 후킹영상 소재 활성화.
```

## 8. 날짜 기본값

이 탭에서 사용하는 날짜 기본값은 사용자의 윈도우 로컬 날짜 기준 `오늘`로 세팅한다.

적용 대상:

- Change Logs 탭의 조회 날짜 기본값
- 새 기록 입력 폼의 `기록일` 기본값

프론트엔드 함수 예시:

```ts
function todayInputValue() {
  const date = new Date();
  date.setMinutes(date.getMinutes() - date.getTimezoneOffset());
  return date.toISOString().slice(0, 10);
}
```

저장 성공 후:

- 기록 textarea는 비움
- 기록일은 다시 오늘로 초기화

백엔드는 `actionDate`가 비어 있을 때만 보조적으로 오늘 날짜를 넣는다. 실제 기본값은 사용자의 PC 날짜를 읽는 프론트엔드 값을 우선한다.

## 9. 데이터 모델 권장안

제품 단위 기록은 광고소재 기록과 의미가 다르므로, 장기적으로는 제품 전용 로그 테이블을 둔다.

권장 Prisma 모델:

```prisma
model ProductChangeLog {
  id          String   @id @default(uuid()) @db.Uuid
  productId   String   @map("product_id") @db.Uuid
  actionDate  DateTime @default(now()) @map("action_date") @db.Date
  text        String   @map("log_text")
  createdBy   String?  @map("created_by") @db.Uuid
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")
  product     Product  @relation(fields: [productId], references: [id], onDelete: Cascade)

  @@index([productId, actionDate])
  @@index([actionDate])
  @@map("product_change_logs")
}
```

`Product` 모델에는 relation을 추가한다.

```prisma
productChangeLogs ProductChangeLog[]
```

## 11. 데이터 집계 기준

제품 목록:

- 선택 날짜 기준 제품별 광고비 합계
- 광고비가 높은 제품부터 표시
- 선택 날짜에 광고 데이터가 없는 제품은 기본적으로 숨김

제품 상세:

- 선택 날짜 기준 해당 제품의 광고 목록 조회
- 광고는 활성화/비활성화로 분리
- 각 광고는 `광고소재명`, `광고비`, `ROAS`만 반환

활성화 기준:

- `adDeliveryStatus` 또는 기존 코드의 `lastStatus`가 `ACTIVE`, `active`면 활성화
- 그 외는 비활성화
- 상태값이 비어 있으면 우선 비활성화 영역에 포함

ROAS:

- 기존 계산값이 있으면 재사용
- 계산 근거가 없으면 `-`로 표시

## 12. 구현 순서

1. Prisma 모델 추가
   - `ProductChangeLog` 모델 추가
   - `Product` relation 추가
   - migration 생성
   - Prisma client regenerate

2. 백엔드 API 추가
   - `GET /change-logs/products`
   - `GET /change-logs/products/:productId`
   - `POST /change-logs/products/:productId/logs`

3. 제품별 성과 집계 함수 작성
   - 선택 날짜 기준 제품별 광고비 집계
   - 선택 제품의 광고 목록 집계
   - 활성화/비활성화 분리
   - ROAS 계산 또는 표시값 구성

4. 프론트 타입 교체
   - `CreativeListItem`, `CreativeDetail` 중심 타입을 제품 중심 타입으로 교체
   - 기존 CSS 클래스는 가능한 재사용

5. 좌측 목록 UI 변경
   - 광고소재 박스를 제품 박스로 변경
   - 제품명과 광고비만 표시
   - 좌측 패널 내부 compact grid 적용

6. 우측 상세 UI 변경
   - 제품명/광고비/광고 수 요약 표시
   - 활성화/비활성화 광고 박스를 같은 줄에 배치
   - 각 광고 표에는 광고소재명, 광고비, ROAS만 표시

7. 기록 UI 변경
   - `새 기록`을 기존 기록 목록 위로 이동
   - 입력 필드는 기록일과 기록 textarea만 유지
   - 기록 목록은 `.log-scroll-box`로 감싸 내부 스크롤 적용

8. 저장 로직 변경
   - `actionType`, `reason`, `memo`, `nextCheckDate` 제거
   - `text` 하나만 저장
   - 저장 성공 후 상세/목록 query invalidate
   - 저장 성공 후 textarea 비움, 기록일은 오늘로 초기화

9. 확인
   - 현재 프로그램과 동일한 좌우 패널 구조인지 확인
   - 좌측 제품 박스가 작고 조밀하게 들어가는지 확인
   - 활성화/비활성화 광고가 같은 줄에 있는지 확인
   - 새 기록이 기록 목록보다 위에 있는지 확인
   - 기록 목록이 일정 높이를 넘으면 내부 스크롤되는지 확인
   - `NOTE`, `KEEP`, `WATCH`, `다음 확인일`, `사유`, `메모`가 사라졌는지 확인

## 14. 완료 기준

- 현재 실행 중인 프로그램과 같은 큰 구조를 유지한다.
- Change Logs의 좌측 목록 단위가 광고 소재가 아니라 제품이다.
- 제품 박스에는 제품명과 광고비만 표시된다.
- 제품 박스는 작고 조밀하며 좌측 패널 안에서 한 줄에 가능한 만큼 들어간다.
- 제품 클릭 시 우측 상세 패널이 선택 제품 기준으로 바뀐다.
- 우측 상세 패널의 활성화 광고와 비활성화 광고 박스가 같은 줄에 배치된다.
- 광고 행에는 광고소재명, 광고비, ROAS만 표시된다.
- 새 기록 입력 영역이 기존 기록 목록보다 위에 있다.
- 기록 목록은 박스 안에 표시되고, 일정 세로 길이 이상이면 내부 스크롤된다.
- 기록 유형 선택 UI가 없다.
- `NOTE`, `KEEP`, `WATCH` 분류가 제품 기록 저장/표시 흐름에 노출되지 않는다.
- `다음 확인일`이 없다.
- `사유`, `메모`가 없고 `기록` 텍스트박스 하나만 있다.
- 날짜 기본값이 윈도우 로컬 날짜 기준 오늘이다.
- 사용자가 기록을 저장하면 선택 제품의 날짜별 기록 목록에 반영된다.

## 15. 같이 생성한 UI 목업

목업 파일:

- `0610_change_logs_product_mockup.html`

목업 목적:

- 실제 기능 구현 전에 현재 실행 중인 프로그램 구조를 유지한 제품 단위 Change Logs 화면을 확인하기 위한 정적 HTML

목업 반영 사항:

- 기존 앱과 같은 사이드바/상단바/좌측 목록/우측 상세 구조
- 좌측 제품 박스는 제품명과 광고비만 표시
- 활성화/비활성화 광고는 같은 줄에 배치
- 새 기록은 기록 목록보다 위에 배치
- 기록 목록은 고정 높이 스크롤 박스로 구성
- 날짜 기본값은 브라우저 로컬 날짜 기준 오늘로 자동 세팅
