# 0610 수정계획 1: Daily Report 탭 신규 설계

## 1. 작업 목적

`Meta-Ads-Performance` 프로그램에 대표 보고용 `Daily Report` 탭을 새로 만든다.

이 탭은 기존 `Ads` 탭처럼 메타 광고 원천 데이터를 광고 소재 단위로 파싱하되, 단순 표나 엑셀 출력이 아니라 대표에게 매일 보여줄 수 있는 커스텀 보고 화면을 제공하는 것이 목적이다.

최종 사용 흐름은 다음과 같다.

1. 사용자가 날짜를 선택한다.
2. 선택 날짜의 소재별 광고 성과가 제품별 박스 단위로 나뉘어 표시된다.
3. 각 제품 박스 안에서 현재 어떤 광고 소재들이 돌아가고 있는지 확인한다.
4. 주요 성과 항목은 선택 날짜 값과 전일자 값을 함께 보여준다.
5. 각 제품 박스 하단에는 해당 날짜의 광고 수정 기록, 즉 Change Logs 성격의 기록이 붙는다.
6. 사용자는 화면을 그대로 인쇄하거나 PDF로 저장해 대표 보고용 자료로 사용한다.

이번 작업은 엑셀 파일 생성 기능이 아니라 앱 내부에서 보고용 화면을 구성하고 브라우저 출력 기능을 붙이는 작업이다.

## 2. 새 탭의 핵심 방향

새 탭 이름은 우선 `Daily Report`로 둔다.

권장 경로는 다음과 같다.

- 프론트 경로: `/daily-report`
- 파일 위치: `apps/web/src/app/daily-report/page.tsx`
- 사이드바 메뉴 라벨: `Daily Report`

이 화면은 기존 `Reports` 탭과 다르다.

기존 `Reports` 탭은 서버에 보고서 파일을 생성하고 다운로드하는 방식이다. 이번 `Daily Report` 탭은 사용자가 앱 화면에서 보고서를 바로 확인하고, `window.print()`와 print CSS로 커스텀 화면을 출력하는 방식이다.

## 3. 현재 코드에서 확인된 구조

주요 파일은 다음과 같다.

  - 실제 layout에서 사용 중인 사이드바

- `apps/web/src/lib/api.ts`
  - `apiGet`, `rangeQuery` 등 API 호출 helper 제공

- `apps/web/src/lib/use-range.ts`
  - 현재 URL과 localStorage 기반 날짜 범위 읽기 hook

- `apps/web/src/lib/date-range.ts`

## 4. Ads 탭에서 그대로 재활용할 수 있는 부분

### 4.1 API 호출 방식

기존 Ads 탭은 아래 API를 사용한다.

```ts
apiGet<CreativePerformanceRow[]>(
  `/metrics/ads/creatives?${rangeQuery(range, { q: query, deliveryStatus: apiDeliveryStatus })}`
)
```

Daily Report도 기본적으로 같은 API를 사용한다.

선택 날짜 데이터를 가져올 때:

```ts
const currentDate = reportDate;
apiGet<CreativePerformanceRow[]>(
  `/metrics/ads/creatives?${rangeQuery({ from: currentDate, to: currentDate }, { q, deliveryStatus })}`
)
```

전일자 데이터를 가져올 때:

```ts
const previousDate = subDays(reportDate, 1);
apiGet<CreativePerformanceRow[]>(
  `/metrics/ads/creatives?${rangeQuery({ from: previousDate, to: previousDate }, { q, deliveryStatus })}`
)
```

단, 현재 `/metrics/ads/creatives` 응답에는 `spendKrw`와 `roas`가 없을 가능성이 높다. 이 항목들은 백엔드 보강이 필요하다.

### 4.2 소재 파싱 로직

기존 백엔드의 `creativeMetrics()`는 이미 아래 작업을 수행한다.

- 광고명에서 제품명 추출
- 광고명에서 소재 번호 추출
- `creativeKey` 기준으로 같은 소재 묶기
- `displayName`, `productName`, `materialNo` 생성
- 활성상태 요약
- 광고비, 구매수, CPA, CTR, CPM 집계

따라서 Daily Report에서 제품별 박스를 만들기 위해 별도의 소재명 파싱 로직을 새로 만들 필요는 없다.

재사용할 핵심 필드:

- `creativeKey`
- `displayName`
- `productName`
- `materialNo`
- `deliveryStatus`
- `totals.spendUsd`
- `totals.purchaseCount`
- `totals.cpaUsd`
- `totals.ctrLinkPct`
- `totals.cpmUsd`
- `dataDays`

### 4.3 날짜 범위 로직

기존 `useRange()`와 `DateRangePicker`를 그대로 활용할 수 있다.

다만 Daily Report는 “매일 대표 보고” 목적이므로 화면 내부에서는 단일 날짜 기준으로 동작하게 하는 것이 좋다.

권장 방식:

- 상단 공통 DateRangePicker는 그대로 둔다.
- Daily Report 페이지에서는 `range.to`를 보고 날짜로 사용한다.
- `from`과 `to`가 다르더라도 Daily Report는 `to` 날짜 하루만 보고 대상으로 삼는다.
- 페이지 상단에 `보고 기준일: YYYY-MM-DD`를 명확히 표시한다.

추후 필요하면 Daily Report 전용 단일 날짜 picker로 분리할 수 있다.

### 4.4 활성상태 필터

Ads 탭의 `DeliveryStatusFilter`를 재사용한다.

```ts
type DeliveryStatusFilter = "active" | "inactive" | "all" | "hasSpend";
```

Daily Report 기본값은 `active`가 적합하다.

단, `hasSpend`는 API에는 `all`로 보내고 프론트에서 `spendUsd > 0`인 행만 남기는 Ads 탭 방식을 그대로 따른다.

### 4.5 검색어 필터

Ads 탭처럼 `q`를 API에 전달할 수 있다.

Daily Report에서도 제품명 또는 소재명 검색은 유지하는 것이 좋다.

다만 대표 보고 화면에서는 검색어 입력 UI가 출력물에 보이면 안 된다. 따라서 화면 조작 UI는 `@media print`에서 숨겨야 한다.

### 4.6 format helper

기존 helper를 재사용한다.

- `money(value, "USD")`
- `money(value, "KRW")`
- `numberFmt(value)`

다만 KRW 광고비와 ROAS까지 들어가면 Daily Report 전용 format helper를 추가해도 된다.

## 5. 신규 화면의 최종 표시 구조

Daily Report 화면은 제품별 박스 구조로 만든다.

전체 구조:

```text
Daily Report
보고 기준일: 2026-06-09

[필터 영역]
- 활성상태
- 검색어
- 표시 항목 선택
- 인쇄 버튼

[요약 KPI 영역]
- 총 광고비 USD
- 총 광고비 KRW
- 총 구매건수
- 전체 CPA
- 전체 ROAS

[제품 박스 1]
제품명
소재별 성과 표
해당 제품의 Change Logs

[제품 박스 2]
제품명
소재별 성과 표
해당 제품의 Change Logs
```

제품 박스는 `productName` 기준으로 그룹화한다.

제품명이 없는 소재는 `"미분류"` 또는 `"제품명 미파싱"` 박스로 묶는다.

## 6. 제품별 박스 설계

각 제품 박스는 하나의 독립된 보고 단위다.

제품 박스 상단:

- 제품명
- 해당 제품의 소재 수
- 해당 제품의 총 광고비 USD
- 해당 제품의 총 광고비 KRW
- 해당 제품의 구매건수
- 해당 제품의 평균 또는 합산 기준 CPA
- 해당 제품의 ROAS

제품 박스 본문:

- 소재별 성과 표

제품 박스 하단:

- 해당 날짜의 Change Logs
- 추후 광고 수정기록 기능이 바뀌어도 이 영역에 붙을 수 있게 독립 컴포넌트로 설계한다.

권장 컴포넌트 분리:

```text
DailyReportPage
  DailyReportToolbar
  DailyReportSummary
  ProductReportBox
    ProductReportHeader
    ProductCreativeTable
    ProductChangeLogSection
```

## 7. Daily Report에 표시할 항목

사용자가 요청한 최종 항목은 다음과 같다.

제품명은 박스 단위로 표시한다.

소재별 표에는 아래 컬럼을 표시한다.

1. 소재
2. 활성상태
3. 집계일수
4. 광고비 USD
5. 광고비 KRW
6. 구매건수
7. CPA
8. CTR
9. CPM
10. ROAS

전일자 데이터 표시 규칙:

- `소재`, `활성상태`, `집계일수`에는 전일자 값을 표시하지 않는다.
- 아래 항목은 현재 값 오른쪽에 전일자 값을 작고 옅은 색으로 표시한다.
  - 광고비 USD
  - 광고비 KRW
  - 구매건수
  - CPA
  - CTR
  - CPM
  - ROAS

예시 UI:

```text
광고비 USD
$123.45  전일 $98.10

구매건수
12  전일 9

ROAS
2.4x  전일 1.8x
```

전일자 표시는 “증감률”이 아니라 “전일 실제값”을 기본으로 한다. 이후 필요하면 증감률 또는 증감 화살표를 추가할 수 있다.

## 8. 현재 API에서 부족한 값과 보강 방향

현재 Ads 탭이 사용하는 `CreativePerformanceRow` 타입은 대략 다음 구조다.

```ts
type CreativePerformanceRow = {
  displayName: string;
  productName: string | null;
  materialNo: string | null;
  deliveryStatus: string | null;
  totals: {
    spendUsd: number;
    purchaseCount: number;
    cpaUsd: number | null;
    ctrLinkPct: number | null;
    cpmUsd: number | null;
  };
  dataDays: number;
};
```

Daily Report에서 추가로 필요한 값:

- `totals.spendKrw`
- `totals.roas`

가능하면 `totals.revenueKrw`도 같이 내려주는 것이 좋다. ROAS 계산 검증과 추후 보고 확장에 유용하다.

권장 확장 타입:

```ts
type CreativePerformanceRow = {
  creativeKey: string;
  displayName: string;
  productName: string | null;
  materialNo: string | null;
  deliveryStatus: string | null;
  totals: {
    spendUsd: number;
    spendKrw: number | null;
    purchaseCount: number;
    cpaUsd: number | null;
    cpaKrw: number | null;
    ctrLinkPct: number | null;
    cpmUsd: number | null;
    roas: number | null;
    revenueKrw: number | null;
  };
  dataDays: number;
};
```

주의할 점:

- 기존 Ads 탭은 현재 `spendUsd`, `purchaseCount`, `cpaUsd`, `ctrLinkPct`, `cpmUsd`만 사용한다.
- 백엔드 응답에 필드를 추가하는 것은 기존 Ads 탭을 깨지 않는다.
- 기존 필드 이름을 바꾸거나 제거하지 말아야 한다.

## 9. 백엔드 로직 수정 계획

수정 대상:

- `apps/api/src/metrics/metrics.service.ts`

핵심 함수:

- `creativeMetrics()`
- `aggregateAdDailyRows()`
- 필요 시 `currentAdMetrics()`

현재 `aggregateAdDailyRows()`는 `MetaAdDailyMetric` 행을 받아 `PeriodMetricCalculator`로 성과를 집계한다.

현재 구조상 광고 소재 단위 집계는 `MetaAdDailyMetric` 기준이라 제품 원가, 환율, 매출, 마진 계산이 완전히 붙어있지 않을 수 있다. 따라서 KRW 광고비와 ROAS를 정확히 계산하려면 아래 중 하나의 전략을 선택한다.

### 9.1 권장 전략 A: MetaAdDailyMetric 기반으로 KRW/ROAS 계산 보강

`MetaAdDailyMetric`에 이미 `productId`, `stage`, `metricDate`, `spendUsd`, `purchaseCount` 성격의 값이 있다.

KRW 광고비:

- 날짜별 환율을 조회한다.
- `spendUsd * rate`로 `spendKrw`를 계산한다.

ROAS:

- 제품별 원가표 또는 판매가 기준이 필요하다.
- 현재 `MetricsService.decorate()`는 adset metric 기준으로 product cost rule과 exchange rate를 조합해 `revenueKrw`, `marginKrw`를 계산한다.
- Daily Report에서도 유사하게 `productId`와 날짜를 기준으로 `ProductCostRule`을 조회해 판매가를 가져온다.
- `revenueKrw = purchaseCount * salePriceKrw`
- `roas = revenueKrw / spendKrw`

장점:

- 기존 creative API에 필요한 값을 직접 붙일 수 있다.
- 제품별 박스와 소재별 row가 자연스럽게 연결된다.

주의:

- productId가 없는 미매칭 소재는 `spendKrw`, `revenueKrw`, `roas`가 `null`일 수 있다.
- 환율이 없는 날짜는 기존 cost rule의 legacy 환율 또는 null 처리 정책을 따라야 한다.

### 9.2 전략 B: 기존 adset decorate 로직 재사용

`decoratedMetrics()`는 `MetaAdsetDailyMetric` 기준으로 KRW, 매출, 마진 계산을 이미 수행한다.

하지만 Daily Report는 소재 단위가 핵심이므로 adset 집계 값을 그대로 쓰면 소재별 ROAS가 부정확해질 수 있다.

따라서 이 전략은 권장하지 않는다.

### 9.3 결론

Daily Report의 소재별 KRW 광고비와 ROAS는 `MetaAdDailyMetric` 기반으로 계산하는 것이 맞다.

구현 시 `aggregateAdDailyRows()`가 다음 필드를 반환하도록 확장한다.

- `spendKrw`
- `revenueKrw`
- `roas`
- 필요 시 `cpaKrw`

단, 기존 호출부가 많으므로 함수 시그니처를 크게 깨지 않는 방식으로 확장한다.

## 10. 전일자 데이터 매칭 로직

Daily Report 프론트는 선택 날짜와 전일자를 각각 호출한다.

```ts
const currentRows = ...
const previousRows = ...
```

전일자 매칭 키 우선순위:

1. `creativeKey`
2. `productName + materialNo`
3. `displayName`

백엔드 응답에는 이미 `creativeKey`가 있으므로 이 값을 우선 사용한다.

매칭 map:

```ts
const previousByCreativeKey = new Map(
  previousRows.map((row) => [row.creativeKey, row])
);
```

현재 row 렌더링 시:

```ts
const previous = previousByCreativeKey.get(row.creativeKey);
```

전일자 값이 없으면 `전일 -`로 표시하거나 아무 것도 표시하지 않는다. 대표 보고 화면에서는 과도한 텍스트를 줄이기 위해 `전일 -`보다 옅은 `-` 표시가 적합하다.

## 11. 제품별 그룹화 로직

프론트에서 currentRows를 제품명 기준으로 그룹화한다.

```ts
function groupRowsByProduct(rows: CreativePerformanceRow[]) {
  const groups = new Map<string, CreativePerformanceRow[]>();
  for (const row of rows) {
    const key = row.productName?.trim() || "제품명 미파싱";
    groups.set(key, [...(groups.get(key) ?? []), row]);
  }
  return Array.from(groups.entries()).map(([productName, rows]) => ({
    productName,
    rows,
    totals: aggregateProductRows(rows)
  }));
}
```

제품 박스 정렬 권장:

1. 광고비 USD 내림차순
2. 구매건수 내림차순
3. 제품명 오름차순

소재 row 정렬 권장:

1. 활성 소재 우선
2. 광고비 USD 내림차순
3. 구매건수 내림차순
4. 소재명 오름차순

## 12. 제품별 합계 계산

제품 박스 상단 요약에 쓸 합계는 프론트에서 계산한다.

```ts
function aggregateProductRows(rows: CreativePerformanceRow[]) {
  const spendUsd = sum(rows.map((row) => row.totals.spendUsd));
  const spendKrw = nullableSum(rows.map((row) => row.totals.spendKrw));
  const purchaseCount = sum(rows.map((row) => row.totals.purchaseCount));
  const revenueKrw = nullableSum(rows.map((row) => row.totals.revenueKrw));

  return {
    spendUsd,
    spendKrw,
    purchaseCount,
    cpaUsd: purchaseCount > 0 ? spendUsd / purchaseCount : null,
    cpaKrw: spendKrw !== null && purchaseCount > 0 ? spendKrw / purchaseCount : null,
    roas: spendKrw && revenueKrw !== null && spendKrw > 0 ? revenueKrw / spendKrw : null
  };
}
```

ROAS 표시 방식은 내부 계산값에 맞춰 하나로 통일한다.

권장:

- `roas = revenue / spend`
- 화면 표시: `2.34x`

만약 기존 대시보드에서 ROAS를 `%`로 표시하고 있다면 전체 앱 기준에 맞춘다.

## 16. Change Logs 확장 설계

사용자는 현재 Change Logs 기능이 원하는 방향이 아니며, 나중에 전면 수정할 계획이다.

따라서 Daily Report는 Change Logs를 강하게 결합하지 말고, 제품별 박스 하단에 붙일 수 있는 확장 슬롯 형태로 설계한다.

권장 컴포넌트:

```tsx
function ProductChangeLogSection({
  productName,
  reportDate
}: {
  productName: string;
  reportDate: string;
}) {
  // 현재는 placeholder 또는 기존 API 연결
  // 추후 제품별 광고 수정 기록 API로 교체
}
```

초기 구현 단계에서는 두 가지 선택지가 있다.

### 16.1 1차 구현: Placeholder만 둔다

각 제품 박스 하단에 아래 형태를 표시한다.

```text
광고 수정 기록
아직 이 날짜에 등록된 기록이 없습니다.
```

이렇게 만들어두면 나중에 Change Logs API가 바뀌어도 Daily Report 레이아웃을 다시 뜯지 않아도 된다.

### 16.2 2차 구현: 기존 Change Logs API 임시 연결

현재 `apps/api/src/change-logs`에 기능이 존재한다.

다만 사용자가 현재 Change Logs 방향을 전부 수정할 예정이라고 했으므로, 기존 API에 맞춰 너무 깊게 구현하지 않는 것이 좋다.

임시 연결을 하더라도 `ProductChangeLogSection` 내부에만 가둔다.

중요 원칙:

- Daily Report 본문 로직과 Change Logs 로직을 섞지 않는다.
- 제품별 박스 컴포넌트는 `changeLogs` prop만 받도록 설계한다.
- 나중에 Change Logs 데이터 모델이 바뀌어도 `ProductChangeLogSection`과 API adapter만 수정하면 되게 한다.

## 19. 백엔드 구현 상세 계획

### 19.1 `aggregateAdDailyRows()` 확장

현재 함수는 광고 row를 `PeriodMetricCalculator`로 집계한다.

여기에 KRW와 매출 계산을 붙이려면 함수가 환율과 제품 원가 rule 정보를 알 수 있어야 한다.

권장 방식:

1. `creativeMetrics()` 안에서 `metrics`를 가져온다.
2. metrics의 `productId`, `metricDate` 목록을 만든다.
3. 필요한 `ProductCostRule`, `ExchangeRate`를 한 번에 조회한다.
4. row별로 `spendKrw`, `revenueKrw` 계산에 필요한 정보를 만든다.
5. `aggregateAdDailyRows()` 또는 새 함수 `aggregateCreativeDailyRows()`에 넘긴다.

새 helper 예시:

```ts
function aggregateCreativeDailyRows(
  calculator: PeriodMetricCalculator,
  rows: Prisma.MetaAdDailyMetricGetPayload<Record<string, never>>[],
  context: CreativeFinancialContext
) {
  ...
}
```

기존 `aggregateAdDailyRows()`를 무리하게 바꾸면 campaign/adset/ads API에 영향이 생길 수 있으므로, Daily Report와 creative API용 새 helper를 만드는 편이 안전하다.

### 19.2 계산 정책

광고비 KRW:

```ts
spendKrw = spendUsd * exchangeRateKrwPerUsd
```

구매건수:

```ts
purchaseCount = adPurchaseCount(row)
```

매출:

```ts
revenueKrw = purchaseCount * salePriceKrw
```

CPA USD:

```ts
cpaUsd = purchaseCount > 0 ? spendUsd / purchaseCount : null
```

CPA KRW:

```ts
cpaKrw = purchaseCount > 0 ? spendKrw / purchaseCount : null
```

ROAS:

```ts
roas = spendKrw > 0 ? revenueKrw / spendKrw : null
```

### 19.3 미매칭 처리

productId가 없거나 원가 rule이 없으면:

- `spendUsd`: 표시 가능
- `purchaseCount`: 표시 가능
- `cpaUsd`: 표시 가능
- `ctrLinkPct`: 표시 가능
- `cpmUsd`: 표시 가능
- `spendKrw`: 환율만 있으면 표시 가능
- `revenueKrw`: null
- `roas`: null

제품명 없는 경우 프론트에서 `제품명 미파싱` 박스로 묶는다.

## 24. 검증 체크리스트

기능 검증:

- `/daily-report` 메뉴가 사이드바에 보인다.
- 날짜 선택 시 해당 날짜 하루 기준으로 데이터가 바뀐다.
- 제품별로 박스가 분리된다.
- 제품명 없는 소재는 별도 박스에 묶인다.
- 소재별 row에 요청한 컬럼이 표시된다.
- 전일자 값이 지정된 컬럼 오른쪽 또는 아래에 작고 옅게 표시된다.
- 소재, 활성상태, 집계일수에는 전일자 값이 표시되지 않는다.
- 표시 항목 선택 시 컬럼이 숨겨지고 다시 보인다.
- 출력 버튼 클릭 시 인쇄 창이 열린다.
- 인쇄 미리보기에서 사이드바, 필터, 버튼이 보이지 않는다.
- 제품 박스가 출력 중간에서 심하게 깨지지 않는다.
- Change Logs 영역이 제품별 박스 하단에 존재한다.

백엔드 검증:

- 기존 Ads 탭이 계속 정상 동작한다.
- `/metrics/ads/creatives` 기존 필드가 유지된다.
- 추가 필드가 null이어도 프론트가 깨지지 않는다.
- 환율 또는 원가 rule이 없는 경우 `roas`가 null 처리된다.

## 26. 최종 목표 화면 요약

최종 화면은 다음 특징을 가져야 한다.

- 대표 보고용으로 바로 보여줄 수 있는 문서형 화면
- 제품별 박스 분리
- 각 제품 박스 안에 현재 운영 중인 광고 소재 목록 표시
- 선택 날짜의 성과와 전일자 성과를 함께 표시
- 제품별 광고 수정 기록 영역 포함
- 사용자가 원하는 항목만 선택해서 볼 수 있음
- 엑셀이 아니라 화면 자체를 출력 또는 PDF 저장 가능

이번 Daily Report 탭은 단순히 Ads 탭의 복사본이 아니라, Ads 탭의 소재 파싱 및 집계 로직을 재사용하되 대표 보고에 맞게 재구성한 제품별 운영 보고 화면이어야 한다.
