# 환율 자동 수집/저장/계산 적용 수정 계획

## 1. 배경

현재 Meta Ads Performance 프로그램은 광고비 원본을 USD로 업로드하고, 원화 광고비와 마진 계산 시 제품 원가 rule에 수동 입력된 `fxRateKrwPerUsd` 값을 사용한다.

현재 구조:

- CSV 업로드: `apps/api/src/uploads/uploads.service.ts`
- 광고비 USD 저장: `MetaAdsetDailyMetric.spendUsd`
- 환율 수동 입력 위치: `ProductCostRule.fxRateKrwPerUsd`
- 환율 적용 계산: `apps/api/src/domain/margin-calculator.ts`
- 지표 계산 조립: `apps/api/src/metrics/metrics.service.ts`
- 화면 입력 위치: `apps/web/src/app/settings/products/page.tsx` 의 "환율" 입력칸

문제:

- 날짜별 실제 환율이 아니라 제품 rule에 저장된 고정 환율을 사용한다.
- 업로드 데이터가 여러 날짜를 포함해도 모든 날짜가 같은 환율로 계산될 수 있다.
- 마진, CPA KRW, ROAS, 광고비 KRW가 날짜별 환율 변동을 반영하지 못한다.

## 2. 사용자 요구사항

사용자가 말한 요구사항:

- 사용자가 프로그램을 실행한 뒤 데이터를 업로드할 때, 업로드 데이터 날짜에 해당하는 환율을 자동으로 가져와야 한다.
- 환율 정보는 API를 통해 받아온다.
- 받아온 환율은 DB에 날짜별로 저장한다.
- 계산이 필요할 때는 DB에 저장된 해당 날짜의 환율값을 사용한다.
- 해당 날짜의 환율 정보가 없는 날에는 가장 최신 환율 정보를 사용한다.

구체화한 정책:

- 기본 환율 대상은 `USD -> KRW`, 즉 `KRW per USD`이다.
- 업로드 CSV 안의 `metricDate` 또는 `dateStart/dateEnd` 기준으로 고유 날짜 목록을 만든다.
- 각 날짜별로 DB에 `USD/KRW` 환율이 없으면 API에서 조회한다.
- API가 해당 날짜 환율을 반환하지 않는 경우, 그 날짜 기준으로 사용 가능한 가장 최신 환율을 사용한다.
- 권장 해석: `요청 날짜 이하의 가장 최근 영업일 환율`을 우선 사용한다. 예: 일요일 데이터면 직전 금요일 환율.
- 만약 요청 날짜 이하 환율이 DB/API 양쪽에 전혀 없으면 provider의 최신 환율을 사용하고, fallback 사유를 DB에 남긴다.
- 외부 API 장애 시 DB에 이미 저장된 가장 최신 환율이 있으면 fallback으로 사용한다.
- 외부 API 장애이고 DB에도 사용할 환율이 없으면 업로드를 실패시키고 명확한 에러를 반환한다.

## 3. 환율 API 후보

### 3.1 1순위: 한국수출입은행 현재환율 API

공식 문서:

- https://www.koreaexim.go.kr/ir/HPHKIR020M01?apino=2&viewtype=C

요청 URL:

```text
https://oapi.koreaexim.go.kr/site/program/financial/exchangeJSON
```

요청 예시:

```text
https://oapi.koreaexim.go.kr/site/program/financial/exchangeJSON?authkey={AUTH_KEY}&searchdate=20260529&data=AP01
```

필수 파라미터:

- `authkey`: 한국수출입은행 Open API 인증키
- `searchdate`: 조회 날짜. 예: `20260529`
- `data`: `AP01` 사용. 환율 조회 타입

사용할 응답 필드:

- `cur_unit === "USD"` 행을 찾는다.
- `deal_bas_r`: 매매기준율. 예: `"1,371.50"`
- 쉼표 제거 후 Decimal/number로 변환한다.

장점:

- 한국 공공기관 공식 환율 API이다.
- USD/KRW 매매기준율을 바로 제공한다.
- 이 프로젝트의 요구사항인 원화 광고비/마진 계산과 잘 맞는다.

주의사항:

- 인증키가 필요하다.
- 문서 기준 일일 호출 제한이 있다. 현재 문서에는 1000회/일 제한이 명시되어 있다.
- 비영업일 또는 영업일 11시 이전 당일 조회 시 null/빈 값이 반환될 수 있다.
- 2025-06-25 이후 요청 도메인이 `oapi.koreaexim.go.kr`로 변경되었다.

환경변수 제안:

```env
KOREA_EXIM_API_KEY="발급받은_키"
EXCHANGE_RATE_PROVIDER="KOREA_EXIM"
EXCHANGE_RATE_BASE_CURRENCY="USD"
EXCHANGE_RATE_QUOTE_CURRENCY="KRW"
```

### 3.2 보조 후보: Frankfurter API

공식 문서:

- https://frankfurter.dev/

특징:

- API key 없이 사용 가능하다.
- 최신/과거 환율 API를 제공한다.
- 여러 중앙은행 데이터를 기반으로 한다.

예시:

```text
https://api.frankfurter.dev/v2/rate/USD/KRW?date=2026-05-29
```

장점:

- 인증키가 없어 개발/테스트가 편하다.
- historical rate 조회가 쉽다.

주의사항:

- 한국 공공기관 고시 환율과 값이 다를 수 있다.
- 사업/정산 기준으로 한국수출입은행 매매기준율을 쓰려면 보조 fallback으로만 쓰는 것이 안전하다.

권장 사용:

- 개발용 fallback 또는 한국수출입은행 API 장애 시 보조 provider로만 사용.
- 실제 운영 기준 환율은 한국수출입은행 `deal_bas_r`을 우선 사용.

### 3.3 한국은행 ECOS API

공식入口:

- https://ecos.bok.or.kr/api/

특징:

- 한국은행 경제통계 시스템 API이다.
- 환율 관련 시계열을 조회할 수 있다.
- 인증키와 통계표/항목 코드 확인이 필요하다.

권장 사용:

- 장기 시계열 분석이나 공식 경제통계 연동이 중요하면 후보가 될 수 있다.
- 현재 프로젝트의 "업로드 날짜별 USD/KRW 매매기준율" 요구에는 한국수출입은행 API가 더 직관적이다.

## 4. DB 설계

### 4.1 Prisma 모델 추가

`apps/api/prisma/schema.prisma`에 다음 모델을 추가한다.

```prisma
enum ExchangeRateFallbackType {
  EXACT
  PREVIOUS_AVAILABLE
  LATEST_AVAILABLE
  MANUAL

  @@map("exchange_rate_fallback_type")
}

model ExchangeRate {
  id              String                   @id @default(uuid()) @db.Uuid
  rateDate        DateTime                 @map("rate_date") @db.Date
  sourceDate      DateTime                 @map("source_date") @db.Date
  baseCurrency    String                   @default("USD") @map("base_currency") @db.VarChar(3)
  quoteCurrency   String                   @default("KRW") @map("quote_currency") @db.VarChar(3)
  rate            Decimal                  @db.Decimal(14, 6)
  provider        String                   @default("KOREA_EXIM")
  providerPayload Json?                    @map("provider_payload")
  fallbackType    ExchangeRateFallbackType @default(EXACT) @map("fallback_type")
  fetchedAt       DateTime                 @default(now()) @map("fetched_at")
  createdAt       DateTime                 @default(now()) @map("created_at")
  updatedAt       DateTime                 @updatedAt @map("updated_at")

  @@unique([rateDate, baseCurrency, quoteCurrency, provider])
  @@index([baseCurrency, quoteCurrency, rateDate])
  @@map("exchange_rates")
}
```

설계 이유:

- `rateDate`: 프로그램이 계산에 쓰려는 날짜. 예: 업로드 row의 metric date.
- `sourceDate`: 실제 API 환율의 기준 날짜. 비영업일 fallback이면 `rateDate`와 다를 수 있다.
- `fallbackType`:
  - `EXACT`: 해당 날짜 환율이 API에서 직접 조회됨.
  - `PREVIOUS_AVAILABLE`: 해당 날짜 환율이 없어 요청일 이전 가장 최근 환율을 사용.
  - `LATEST_AVAILABLE`: 요청일 이전 환율도 없어서 provider 최신 환율을 사용.
  - `MANUAL`: 향후 수동 입력/보정용.

중요:

- 비영업일에도 `rateDate`별 row를 만든다.
- 예: 2026-05-24 일요일 데이터가 있고 2026-05-22 금요일 환율을 사용하면:
  - `rateDate = 2026-05-24`
  - `sourceDate = 2026-05-22`
  - `fallbackType = PREVIOUS_AVAILABLE`

이렇게 해야 계산 시 단순히 `metricDate`로 환율을 찾을 수 있고, 실제 출처도 추적할 수 있다.

### 4.2 기존 ProductCostRule 환율 필드 처리

현재 `ProductCostRule.fxRateKrwPerUsd`는 필수 필드이다.

변경 방향:

- 계산 로직에서는 더 이상 이 필드를 우선 사용하지 않는다.
- 신규 원가 rule 생성 시 이 값은 legacy fallback으로만 유지한다.
- UI에서는 환율 입력칸을 제거하거나 "기존 fallback 환율" 정도로 숨긴다.
- DB 필드는 당장 삭제하지 말고, 마이그레이션 리스크를 줄이기 위해 보존한다.
- 신규 rule 생성 API에서는 `fxRateKrwPerUsd`가 없어도 생성 가능하게 한다.
  - 예: 누락 시 `0` 또는 최근 저장 환율을 넣는다.
  - 단, 계산에는 `ExchangeRate.rate`를 사용한다.

추천:

- 1차 구현에서는 컬럼을 유지하고 service에서 default 값을 채운다.
- 안정화 후 별도 마이그레이션에서 nullable 또는 제거를 검토한다.

### 5.2 날짜별 환율 확보 흐름

업로드 시:

1. CSV를 파싱한다.
2. `parsedRow.metricDate` 목록을 만든다.
3. 고유 날짜만 추린다.
4. `exchangeRatesService.ensureUsdKrwRatesForDates(uniqueDates)`를 호출한다.
5. 누락된 날짜만 provider API를 호출한다.
6. 각 날짜별 `ExchangeRate` row를 DB에 `upsert`한다.
7. 이후 기존 업로드 row/metric 저장을 진행한다.

주의:

- row마다 API를 호출하지 않는다.
- 날짜별 1회만 조회한다.
- 같은 날짜가 이미 DB에 있으면 외부 API를 호출하지 않는다.

### 5.3 API 호출 제한 대응

한국수출입은행 API는 일일 호출 제한이 있으므로:

- 업로드 파일의 unique date만 조회한다.
- 이미 DB에 있는 날짜는 재조회하지 않는다.
- 같은 업로드 안에서 Promise 병렬 호출을 과도하게 하지 않는다.
- 필요하면 3~5개 단위 concurrency 제한을 둔다.
- provider result가 일일 제한 오류이면 업로드를 중단하고 안내 에러를 반환한다.

### 5.4 fallback 정책

날짜 `D`에 대해:

1. DB에 `rateDate = D`인 USD/KRW 환율이 있으면 사용.
2. 없으면 provider에서 `D` 정확 조회.
3. 정확 조회 성공:
   - `rateDate = D`
   - `sourceDate = D`
   - `fallbackType = EXACT`
4. 정확 조회 실패 또는 빈 값:
   - DB에서 `sourceDate <= D` 또는 `rateDate <= D` 중 가장 최신 row를 찾는다.
   - 있으면 그 값을 사용하여 `rateDate = D`, `sourceDate = found.sourceDate`, `fallbackType = PREVIOUS_AVAILABLE`으로 저장한다.
5. DB에도 없으면 provider에서 과거 날짜를 역순 조회한다.
   - 예: D-1, D-2, ..., D-10
   - 성공하면 `PREVIOUS_AVAILABLE`.
6. 그래도 없으면 provider 최신 환율을 조회한다.
   - 성공하면 `rateDate = D`, `sourceDate = latest.date`, `fallbackType = LATEST_AVAILABLE`.
7. 그래도 실패하면 업로드 실패.

사용자 요구사항 문구 반영:

- "환율 정보가 없는 날엔 가장 최신 환율 정보"를 사용한다.
- 단, 과거 업로드의 정확성을 위해 요청일 기준 이전 최신 환율을 먼저 사용하고, 그마저 없을 때 provider 최신 환율을 사용한다.

## 6. 업로드 로직 변경

파일:

- `apps/api/src/uploads/uploads.service.ts`

변경점:

- `UploadsModule`에서 `ExchangeRatesModule`을 import한다.
- `UploadsService` 생성자에 `ExchangeRatesService` 주입.
- CSV 파싱 후 row 처리 루프 전에 환율 확보:

```ts
const parsedRows = rows.map(...);
const metricDates = parsedRows
  .map((row) => row.parsedRow?.metricDate)
  .filter(Boolean);
const exchangeRates = await this.exchangeRatesService.ensureUsdKrwRatesForDates(metricDates);
```

주의:

- 현재 코드는 parse와 DB insert를 한 루프에서 처리한다.
- 환율 날짜를 선확보하려면 parse 결과를 배열로 보관하거나, 첫 루프에서 parse만 수행하고 두 번째 루프에서 DB 저장하는 구조가 필요하다.
- 헤더 오류가 있으면 환율 API 호출 전에 실패시키는 것이 좋다.

const spendKrw = input.spendUsd * exchangeRateKrwPerUsd;

export type MarginMetricInput = {
  spendUsd: number;

### 7.2 MetricsService.decorate

- productId 목록으로 costRules/cpaRules를 가져온다.

ruleStatus:

## 9. 선택: 환율 관리 화면/API

- `GET /api/exchange-rates?from=YYYY-MM-DD&to=YYYY-MM-DD`
- `POST /api/exchange-rates/sync`
- `PATCH /api/exchange-rates/:id` 수동 보정

화면:

## 10. 테스트 계획

### 10.1 단위 테스트

파일:

- `apps/api/src/domain/domain.spec.ts`
- 신규 `apps/api/src/exchange-rates/exchange-rates.service.spec.ts`

테스트:

- 정확 날짜 환율이 있으면 `EXACT`로 저장.
- provider가 빈 배열을 반환하면 직전 저장 환율을 사용.
- 직전 저장 환율이 없으면 provider에서 과거 날짜를 lookback.
- 과거 lookback도 실패하면 latest를 사용.
- 쉼표가 포함된 `deal_bas_r` 문자열을 Decimal로 정상 파싱.
- `spendUsd * exchangeRate`로 spendKrw 계산.
- costRule의 legacy `fxRateKrwPerUsd`가 있어도 ExchangeRate가 우선.

### 10.2 업로드 테스트

테스트:

- 같은 날짜 row가 여러 개 있어도 API 호출은 날짜당 1회.
- 업로드 파일에 주말 날짜가 있으면 fallback row가 생성됨.
- API 장애 + DB latest 있음: DB latest로 fallback.
- API 장애 + DB latest 없음: 업로드 실패.

### 10.3 수동 검증

검증 순서:

1. DB migration 실행.
2. API key 환경변수 설정.
3. 서버 실행.
4. CSV 업로드.
5. `exchange_rates` 테이블에 날짜별 row가 생성되는지 확인.
6. 대시보드에서 `spendKrw`, `cpaKrw`, `marginKrw`가 환율 반영 계산인지 확인.
7. 주말/공휴일 데이터가 직전 또는 최신 환율 fallback으로 계산되는지 확인.

## 11. 마이그레이션 작업 순서

1. Prisma schema에 `ExchangeRateFallbackType`, `ExchangeRate` 추가.
2. `prisma migrate dev --schema prisma/schema.prisma` 실행.

4. 한국수출입은행 provider 구현.

7. ProductCostRule 생성 API에서 `fxRateKrwPerUsd` 필수 제거.
8. Product Settings UI에서 환율 입력칸 제거.
9. Health/Report에 `missingExchangeRateCount` 추가 검토.
10. 테스트 작성 및 실행.

## 12. 환경변수/README 업데이트

`apps/api/.env.example` 또는 README에 추가:

```env
KOREA_EXIM_API_KEY=""
EXCHANGE_RATE_PROVIDER="KOREA_EXIM"
EXCHANGE_RATE_LOOKBACK_DAYS=10
```

주의:

- 실제 API key는 git에 커밋하지 않는다.
- API key가 없으면 업로드 시 환율 자동 동기화를 실패시키거나, 개발환경에서는 Frankfurter fallback을 허용할 수 있다.

## 14. 구현 난이도 메모

작업량은 중간급이다.

- 단순 구현: 반나절~하루
- 안정 구현: 1~2일

작업량이 늘어나는 이유:

- DB migration이 필요하다.
- 외부 API 연동과 API key 환경변수 처리가 필요하다.
- 업로드 흐름에 날짜별 사전 동기화가 들어간다.
- 계산 로직이 cost rule 기반 환율에서 날짜별 환율 기반으로 바뀐다.
- 비영업일/공휴일/당일 11시 이전/API 장애 fallback 정책을 테스트해야 한다.

그래도 구조는 명확하다. 새 기능의 핵심은 `ExchangeRatesService.ensureUsdKrwRatesForDates()`를 안정적으로 만들고, `MetricsService.decorate()`에서 그 환율을 사용하게 바꾸는 것이다.
