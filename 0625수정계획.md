## 1. 현재 상황 요약

현재 쿠팡 업로드 탭에는 4개 업로드 칸이 있다.

```text
Sales Analysis XLSX
Ads Performance XLSX
Product Margin CSV
Sale Price Text
```

각 업로드의 현재 역할은 다음과 같다.

```text
Sales Analysis XLSX
- 쿠팡 판매 분석 엑셀.
- 판매일, 상품명/옵션명, 매출, 취소, 판매수량 등을 파싱한다.
- CoupangSaleLine에 저장된다.
- 순수익 계산의 매출 기준은 이 파일의 netSalesKrw다.

Ads Performance XLSX
- 쿠팡 광고 성과 엑셀.
- 광고비, 광고 전환 매출, 전환 수량 등을 파싱한다.
- CoupangAdMetric에 저장된다.

Product Margin CSV
- 쿠팡 상품별 원가/비용 기준 CSV.
- 현재 구현은 이 CSV의 판매가도 CoupangCostRule.salePriceKrw에 저장하고 있다.
- 이 부분이 문제다.

Sale Price Text
- 원본 판매가 보조 자료.
- 상품명과 가격만 있는 txt를 파싱해서 기존 cost rule의 비용값을 복사하고 salePriceKrw만 바꾸는 방식이다.
```

현재 쿠팡 순수익 계산 자체는 판매가를 곱해서 매출을 만들지 않는다. 매출은 `Sales Analysis XLSX`에서 온 `netSalesKrw`를 사용한다. 따라서 CSV의 판매가가 프로모션가여도 현재 순매출 자체를 직접 망치지는 않는다.

하지만 현재 CSV의 판매가가 다음 위치에 저장/표시된다.

```text
apps/api/src/coupang/coupang.service.ts
- importMarginCsv에서 CoupangCostRule.salePriceKrw = row.parsedRow.salePriceKrw로 저장

apps/api/src/coupang/coupang.service.ts
- buildProductProfitRows에서 row.salePriceKrw = costRule.salePriceKrw로 반환

apps/api/src/coupang/coupang.service.ts
- dailyReport에서 salePriceKrw를 엑셀/화면으로 반환

apps/web/src/app/coupang/daily-report/page.tsx
- Daily Report의 판매가 컬럼에 표시

apps/web/src/app/coupang/products/page.tsx
- Product Settings 테이블의 Sale Price에 표시
```

즉 현재 문제는 다음과 같다.

```text
Product Margin CSV의 판매가가 프로모션 적용 가격이면,
원본 판매가로 보여야 할 값이 프로모션 가격으로 저장/표시될 수 있다.
```

또 하나의 주의점이 있다.

```text
Product Margin CSV의 판매수수료 금액이 판매가 기반으로 계산된 값이라면,
프로모션가 기준 판매수수료가 비용 계산에 들어갈 가능성이 있다.
현재 calculateCoupangProfit 호출은 feeMode: "PER_UNIT"을 사용하므로 salesFeeKrw * 수량으로 계산한다.
CSV에 판매수수료율이 안정적으로 들어온다면 salesFeeRate * netSalesKrw 방식이 더 안전하다.
```

이 부분은 이번 수정에서 반드시 같이 점검해야 한다.

## 2. 사용자가 원하는 방향

사용자가 원하는 방향은 명확하다.

```text
1. Product Margin CSV에 있는 판매가는 쓰면 안 된다.
2. Product Margin CSV는 원가/수수료/배송비/반품비 같은 비용 기준만 저장해야 한다.
3. 원본 판매가는 Sale Price Text 또는 상품 설정에서 따로 관리한다.
4. 매일 또는 필요 시 프로모션 파일을 업로드한다.
5. 프로모션 파일의 등록상품명/옵션명을 기존 쿠팡 포함 키워드/제외 키워드 규칙으로 다시 분류한다.
6. 사용자가 선택한 날짜가 프로모션 시작일시와 종료일시 사이에 있을 때만 프로모션 가격을 적용한다.
7. 프로모션 가격은 기준 원가 rule에 저장하면 안 된다.
8. 프로모션 가격은 날짜별 표시/적용용 오버레이로만 사용한다.
```

절대 하지 말아야 할 것:

```text
Product Margin CSV의 판매가를 CoupangCostRule.salePriceKrw에 덮어쓰기 금지.
프로모션 가격을 CoupangCostRule.salePriceKrw에 덮어쓰기 금지.
프로모션 파일의 판매가/프로모션 가격으로 Sales Analysis의 순매출을 재계산 금지.
프로모션 가격을 기준 판매가처럼 영구 저장 금지.
```

매출의 권위 source는 계속 `Sales Analysis XLSX`다.

```text
순매출 = Sales Analysis XLSX에서 파싱한 netSalesKrw
판매가 표시 = 날짜별 가격 resolver가 결정한 표시 판매가
비용 = Product Margin CSV와 상품 설정에서 관리하는 원가/수수료/배송비/반품비 기준
광고비 = Ads Performance XLSX
```

## 4. 구현 설계 개요

이번 수정은 4개 축으로 나눈다.

```text
1. Product Margin CSV에서 판매가 저장 제거
2. 원본 판매가와 프로모션 판매가의 역할 분리
3. 프로모션 XLSX 업로드/파싱/매칭/저장 추가
4. 날짜별 가격 resolver를 통해 daily report와 product profit에 표시 가격 적용
```

권장 설계:

```text
CoupangCostRule.salePriceKrw는 당장 스키마에서 제거하지 않는다.
기존 코드 영향이 있으므로 "원본 판매가/base price 보관용"으로 유지한다.
단, Product Margin CSV에서는 이 값을 절대 새로 읽어오지 않는다.

Sale Price Text는 원본 판매가를 갱신하는 source로 유지한다.
Product Settings에서 직접 입력하는 salePriceKrw도 원본 판매가 source로 유지한다.

Promotion XLSX는 별도 모델에 저장한다.
프로모션가는 cost rule이 아니라 promotion table에 저장한다.
```

이 방식은 기존 DB/화면과 충돌이 적고, 사용자의 의도인 “프로모션가는 날짜별 적용값이지 기준 판매가가 아니다”를 지킨다.

### 7.2 importMarginCsv 수정

현재:

```ts
await tx.coupangCostRule.create({
  data: {
    coupangProductId: product.id,
    salePriceKrw: decimal(row.parsedRow.salePriceKrw),
    ...
  }
});
```

변경 방향:

```text
1. 기존 최신 cost rule이 있으면 기존 salePriceKrw를 복사한다.
2. 기존 최신 cost rule이 없으면 salePriceKrw는 0으로 저장하되, 표시할 때는 0을 null처럼 처리한다.
3. Product Margin CSV의 판매가는 절대 사용하지 않는다.
4. 가능하면 warning row error를 남겨 "원본 판매가가 아직 없다"를 알려준다.
```

예시 로직:

```ts
const latestCostRule = await tx.coupangCostRule.findFirst({
  where: { coupangProductId: product.id },
  orderBy: [{ effectiveFrom: "desc" }, { createdAt: "desc" }]
});

const preservedSalePriceKrw = latestCostRule?.salePriceKrw ?? decimal(0);

await tx.coupangCostRule.create({
  data: {
    coupangProductId: product.id,
    salePriceKrw: preservedSalePriceKrw,
    supplyPriceKrw: decimal(row.parsedRow.supplyPriceKrw),
    productCostKrw: decimal(row.parsedRow.productCostKrw),
    ...
  }
});
```

주의:

```text
margin CSV를 price text보다 나중에 올려도 원본 판매가가 사라지면 안 된다.
margin CSV를 먼저 올린 경우에는 원본 판매가가 0 또는 null 상태일 수 있다.
이후 price text를 올리면 기존 비용 rule을 복사하면서 salePriceKrw만 채우는 현재 방식과 잘 맞는다.
```

## 8. 판매수수료 계산 주의사항

현재 `buildProductProfitRows`는 다음처럼 계산기를 호출한다.

```ts
const calculated = calculateCoupangProfit(
  sales,
  costInput(costRule),
  ads,
  {
    feeMode: "PER_UNIT",
    includeReturnCost: true,
    useGrowthCost: true
  }
);
```

문제:

```text
Product Margin CSV의 판매수수료 금액이 프로모션가 기반이면,
salesFeeKrw * 판매수량 계산이 부정확할 수 있다.
```

권장 수정:

```text
salesFeeRate가 0보다 크면 RATE 방식 사용.
salesFeeRate가 없거나 0이면 기존 PER_UNIT 방식 fallback.
```

예시:

```ts
const feeMode = numberFrom(costRule.salesFeeRate) > 0 ? "RATE" : "PER_UNIT";

const calculated = calculateCoupangProfit(
  sales,
  costInput(costRule),
  ads,
  {
    feeMode,
    includeReturnCost: true,
    useGrowthCost: true
  }
);
```

`RATE` 방식은 이미 `calculateCoupangProfit`에 구현되어 있다.

```ts
options.feeMode === "RATE"
  ? netSalesKrw * finiteNumber(cost.salesFeeRate)
  : finiteNumber(cost.salesFeeKrw) * quantity;
```

이 변경은 판매가 자체를 쓰는 것이 아니라 실제 순매출 `netSalesKrw`에 수수료율을 곱하는 방식이다. 프로모션으로 실제 매출이 낮아져도 Sales Analysis의 순매출이 이미 낮아져 있으므로 더 안전하다.

테스트 추가:

```text
salesFeeRate가 있으면 netSalesKrw * rate로 수수료 계산
salesFeeRate가 0이면 salesFeeKrw * quantity fallback
```

서비스 메서드:

```ts
async importPromotionXlsx(file: Express.Multer.File | undefined, body: Record<string, unknown>) {
  const upload = this.assertFile(file, "Coupang promotion XLSX file is required.");
  const parsed = this.promotionParser.parseBuffer(upload.buffer);
  const batch = await this.createBatch({
    sourceType: CoupangUploadSourceType.PROMOTION,
    ...
  });

  const rules = await this.matcherRules();

  await this.prisma.$transaction(async (tx) => {
    for (const row of parsed.rows) {
      if (!row.parsedRow) {
        createRowErrors(...);
        continue;
      }

      const match = this.matcher.matchText(
        row.parsedRow.productText,
        rules,
        row.parsedRow.promotionStartDate
      );

      await tx.coupangPromotionPrice.create({
        data: {
          uploadBatchId: batch.id,
          rowNumber: row.rowNumber,
          sourceRowHash: row.sourceRowHash,
          ...
          coupangProductId: match.productId,
          coupangProductRuleId: match.matchRuleId,
          matchSource: match.source === "RULE" ? MatchSource.RULE : MatchSource.UNMATCHED,
          validationStatus: statusFromPromotionMatch(match),
          validationErrors: match.reason === "MATCHED" ? [] : [ ... ],
          rawRow: row.rawRow
        }
      });

      if (match.reason !== "MATCHED") {
        createRowErrors(... WARNING ...);
      }
    }
  });
}
```

반환값:

```ts
return {
  batchId: batch.id,
  schemaVersion: COUPANG_PROMOTION_SCHEMA_VERSION,
  rowCount,
  validRowCount,
  warningCount,
  errorCount,
  matchedCount,
  unmatchedCount,
  dataStart,
  dataEnd
};
```

`dataStart`, `dataEnd`는 프로모션 시작일/종료일 범위로 batch에 저장한다.

## 12. 가격 resolver 설계

필요한 기능:

```text
상품별 기준 판매가(base price)
상품별 날짜 활성 프로모션 가격(promotion price)
최종 표시 판매가(resolved sale price)
가격 source
```

권장 타입:

```ts
type CoupangResolvedPrice = {
  salePriceKrw: number | null;
  baseSalePriceKrw: number | null;
  promotionPriceKrw: number | null;
  priceSource: "PROMOTION" | "BASE" | "MISSING" | "CONFLICT";
  warning?: string;
};
```

적용 규칙:

```text
1. 선택 날짜에 활성인 프로모션이 정확히 1개 있으면 promotionPriceKrw 사용.
2. 활성 프로모션이 여러 개이고 가격이 모두 같으면 그 가격 사용.
3. 활성 프로모션이 여러 개이고 가격이 다르면 CONFLICT로 처리.
4. CONFLICT일 때는 잘못된 값을 보여주지 않기 위해 baseSalePriceKrw로 fallback하고 warning을 남긴다.
5. 활성 프로모션이 없으면 baseSalePriceKrw 사용.
6. baseSalePriceKrw가 0 또는 null이면 salePriceKrw는 null로 반환.
```

중요:

```text
프로모션 가격 resolver는 표시 판매가를 결정한다.
Sales Analysis의 netSalesKrw를 바꾸지 않는다.
```

쿼리 기준:

```ts
where: {
  coupangProductId: { in: productIds },
  validationStatus: { not: RowValidationStatus.ERROR },
  promotionStartDate: { lte: reportDate },
  promotionEndDate: { gte: reportDate }
}
```

단, `UNMATCHED`는 productId가 없으므로 자연스럽게 제외된다. `WARNING`은 매칭은 되었지만 보조 경고가 있는 경우만 적용할지 결정해야 한다. 권장:

```text
productId가 있고 promotionPriceKrw가 유효하면 적용 후보.
단, validationErrors에 INVALID_STATUS나 PRICE_CONFLICT 같은 중대 경고가 있으면 제외.
```

`deleteUpload`에서 sourceType이 PROMOTION이면 해당 batch의 `coupangPromotionPrice` rows가 cascade로 지워져야 한다.

스키마 relation에 `onDelete: Cascade`를 넣으면 batch 삭제 시 자동 삭제된다.

확인할 것:

```text
CoupangUploadBatch -> CoupangPromotionPrice relation onDelete Cascade
deleteUpload에서 별도 deleteMany가 필요한지 Prisma relation 동작 확인
```

```text

1. 같은 가격이면 적용
2. 다른 가격이면 최신 uploadedAt batch의 값만 쓸지, conflict로 볼지 결정 필요
```

사용자 의도상 잘못된 가격을 조용히 쓰는 것이 위험하므로 초기 구현은 conflict로 보고 base price fallback을 권장한다.

## 22. 사용자 확인이 필요한 결정

다음 세션에서 구현 전에 가능하면 확인하면 좋은 점:

```text
1. Daily Report 엑셀에 판매가구분/원본판매가/프로모션판매가 컬럼을 추가해도 되는가?
2. 기간 조회 화면에서 판매가는 range.to 날짜 기준이면 되는가?
3. 같은 상품에 같은 날짜의 프로모션 가격이 여러 개 다르게 잡히면 base price fallback + warning이 맞는가?
4. 판매수수료는 salesFeeRate가 있으면 netSalesKrw * rate로 바꾸는 것이 맞는가?
```

하지만 사용자가 바로 구현을 원하면 다음 기본값으로 진행한다.

```text
Daily Report 기본 판매가 컬럼은 resolved sale price만 표시.
API에는 baseSalePriceKrw/promotionPriceKrw/priceSource를 같이 내려준다.
기간 조회 판매가는 range.to 기준.
프로모션 가격 충돌은 base fallback + warning.
판매수수료는 salesFeeRate > 0이면 RATE, 아니면 PER_UNIT.
```

## 23. 최종 의도 한 줄 요약

