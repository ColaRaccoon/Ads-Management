# 0611 수정계획 - Meta-Ads-Performance에 카페24 실제 판매 데이터 기반 제품 성과 기능 추가

## 0. 문서 목적

이 문서는 다음 작업 세션에서 이 파일 하나만 읽어도 요구사항, 기존 프로그램 흐름, 참고할 파이썬 로직, 실제 CSV 구조, 신규 DB/API/UI 설계, 테스트 기준을 정확히 이어받아 구현할 수 있도록 작성한 작업 지침서다.

이번 작업의 핵심은 기존 `Meta-Ads-Performance` 웹앱에 카페24 주문 CSV 업로드 기능을 추가하고, 기존 Meta 광고 관리자 CSV에서 들어온 광고비 소진값과 카페24의 실제 판매 데이터를 제품 단위로 대조하여 “소재별/광고세트별 성과”뿐 아니라 “제품별 실제 성과”도 볼 수 있게 하는 것이다.

주의: 이 문서는 계획서다. 이 문서 작성 시점에는 코드 수정 작업을 하지 않았다.

## 5. 기존 파이썬 카페24 매칭 로직

참고 파일:

```text
C:\Users\seong\Desktop\계산기\Meta-Margin-Calculator\core\cafe24_matcher.py
```

현재 핵심 동작:

```text
1. Cafe24OrderRecord 1행을 Cafe24RowConfig 목록과 비교한다.
2. _matches(record, row)가 true인 row들을 후보로 모은다.
3. 후보 row.name 중복을 제거한다.
4. 후보가 1개면 MATCHED
5. 후보가 0개면 NO_MATCH
6. 후보가 2개 이상이면 AMBIGUOUS_MATCH
```

매칭 조건:

```text
1. row.product_numbers가 있으면 record.product_no가 product_numbers에 포함되어야 한다.
2. row.product_name_aliases가 있으면 주문상품명 또는 주문상품명(옵션포함)에 alias가 포함되어야 한다.
3. row.option_include_keywords가 있으면 주문상품명(옵션포함)에 모든 keyword가 포함되어야 한다.
4. row.option_exclude_keywords가 있으면 주문상품명(옵션포함)에 어떤 keyword도 포함되면 안 된다.
```

정규화:

```python
def normalize(text: str) -> str:
    return "".join(str(text).split()).lower()
```

상품번호 정규화:

```python
def _normalize_product_no(value: str) -> str:
    text = str(value).strip()
    if text.endswith(".0"):
        return text[:-2]
    return text
```

이 로직은 TypeScript/Nest 도메인 함수로 거의 그대로 이식하면 된다.

권장 신규 파일:

```text

apps/api/src/domain/cafe24-matcher.ts
apps/api/src/domain/cafe24-sales-calculator.ts
```

이번 요구사항 기준 권장:

- 실제 제품별 성과가 목적이므로 `버닝 웨이브바`와 `버닝 웨이브바 1+1`을 제품 성과에서 함께 묶어 볼 수 있어야 한다.

최소 구현안:

- `Cafe24ProductRule`에 아래 override 필드를 둔다.


- 현재 요구사항은 정상 주문 판매수 기준이므로 음수 발생 시 warning/error로 분리하는 것을 권장한다.

날짜 파싱:

- `발주일`은 보통 `2026-06-11 10:00:00` 형태다.
- `orderDate`는 `YYYY-MM-DD` date only로 저장한다.
- 기존 `toDateOnly`, `parseDateString`은 시간 포함 문자열 처리가 제한적이므로 카페24용 datetime parser가 필요하다.

중복 기준:

- 같은 업로드 파일 내 중복 판정은 `주문번호 + 품목별 주문번호 + 상품번호 + 옵션포함명` 조합을 우선 사용한다.
- DB unique까지 강하게 걸지는 말고, 배치 내 중복 warning 정도로 시작해도 된다.

## 10. 카페24 매칭 구현 지침

권장 파일:

```text

```

파이썬 로직을 그대로 이식한다.

TypeScript 형태 예시:

```ts
export type Cafe24RuleInput = {
  id: string;
  productId: string;
  productNumbers: string[];
  productNameAliases: string[];
  optionIncludeKeywords: string[];
  optionExcludeKeywords: string[];
  priority: number;

  productId: string | null;
  source: "RULE" | "UNMATCHED";
  matchRuleId: string | null;
  reason: "MATCHED" | "NO_MATCH" | "AMBIGUOUS_MATCH";
  candidates: string[];
};
```

매칭 순서:

```text
1. active rule만 필터링
2. orderDate가 validFrom/validTo 범위 안에 있는 rule만 필터링
3. priority 오름차순으로 정렬
4. 각 rule에 대해 아래 조건 확인
   - productNumbers 조건
   - productNameAliases 조건
   - optionIncludeKeywords 조건
   - optionExcludeKeywords 조건
5. 매칭된 rule의 displayName 또는 id 기준으로 unique 후보 구성
6. 후보 1개면 MATCHED
7. 후보 0개면 NO_MATCH
8. 후보 2개 이상이면 AMBIGUOUS_MATCH
```

정규화:

```ts
export function normalizeCafe24Text(value: string) {
  return String(value).split(/\s+/).join("").toLowerCase();
}
```

상품번호 정규화:

```ts
export function normalizeProductNo(value: string) {
  const text = String(value).trim();
  return text.endsWith(".0") ? text.slice(0, -2) : text;
}
```

확정된 운영 규칙:

- 옵션명에 `+`가 들어간 웨이브바 옵션은 1+1이다.
- 별도의 판매가 조건은 필요 없다.
- 따라서 파이썬의 `option_include_keywords: ["+"]`, `option_exclude_keywords: ["+"]` 방식 그대로 사용한다.

## 12. 실제 성과 계산식

제품/카페24 룰별 row 계산:

```text
quantity = 카페24 주문 행 수량 합계
revenueKrw = 실제 판매가 기준 매출
```

매출 계산 방식 선택:

1. 카페24 CSV의 `판매가 * 수량`을 사용
2. Product Setting 또는 Cafe24ProductRule override의 판매가 * 수량을 사용
3. 카페24 CSV의 `총 결제금액`을 사용

이번 요구사항 기준 추천:

- 판매수는 카페24 CSV의 `수량` 합계 기준이다.
- 매출은 기존 파이썬 프로그램과 동일하게 설정 판매가 * 수량을 우선 사용한다.
- 단, 실제 판매가와 설정 판매가가 다르면 warning으로 보여주는 것이 좋다.
- `총 결제금액`은 결제수단/선불금/쿠폰 등으로 인해 0 또는 할인 반영값이 될 수 있으므로 판매수 기준에서 제외한다.

기존 파이썬 계산식:

```text
vat = price * vat_rate
ad_cost_krw = Meta 광고비 USD 합계 * 환율
total_cost = ((vat + cost + shipping) * quantity) + ad_cost_krw
revenue = price * quantity
roas = 광고비가 0이면 null, 아니면 revenue / ad_cost_krw * 100
margin = revenue - total_cost
```

웹앱 권장 계산식:

```text
vatKrw = salePriceKrw * 0.1 또는 ProductCostRule.vatKrw
unitGrossCostKrw = vatKrw + productCostKrw + shippingKrw + extraCostKrw
grossCostKrw = unitGrossCostKrw * quantity
adSpendKrw = 날짜별 Meta spendUsd * exchangeRate 합계
totalCostKrw = grossCostKrw + adSpendKrw
revenueKrw = salePriceKrw * quantity
marginKrw = revenueKrw - totalCostKrw
roas = adSpendKrw === 0 ? null : revenueKrw / adSpendKrw
cpaKrw = quantity === 0 ? null : adSpendKrw / quantity
marginRate = revenueKrw === 0 ? null : marginKrw / revenueKrw
```

ROAS 표시 단위:

- 기존 파이썬 xlsx는 `* 100` 형태로 퍼센트 숫자를 만들었다.
- 현재 웹앱 `MarginCalculator`는 `revenueKrw / spendKrw` 비율을 반환한다.
- 웹 UI 표시 방식과 맞춰야 한다. 기존 웹앱이 `roas`를 ratio로 쓰고 있다면 신규 API도 ratio로 통일한다.

## 13. ROAS 그룹과 광고비 배정

파이썬 기존 개념:

```text
ad_cost_source
roas_group
```

용도:

- 단품 행에 광고비를 배정한다.
- 1+1 행에는 광고비를 0으로 두되, ROAS 계산 시 단품 + 1+1 매출을 같은 그룹으로 묶는다.

예:

```text
버닝 웨이브바
- ad_cost_source: 버닝 웨이브바
- roas_group: burning_wavebar

버닝 웨이브바 1+1
- ad_cost_source: ""
- roas_group: burning_wavebar
```

웹앱에서의 구현 선택:

### 선택 A: 파이썬 방식을 그대로 재현

장점:

- 기존 xlsx와 거의 같은 결과를 낸다.
- 사용자가 익숙한 계산 방식이다.

단점:

- 제품 단위 실제 성과 API에서 단품/1+1 row와 제품 그룹 row가 섞일 수 있다.

### 선택 B: 제품 단위로 광고비와 매출을 모두 합산

장점:

- “제품별 실제 성과” 목적에는 더 자연스럽다.
- 버닝 웨이브바 단품 + 1+1을 하나의 제품 성과로 보여줄 수 있다.

단점:

- 파이썬 xlsx의 row별 광고비 배정 방식과 숫자가 다르게 보일 수 있다.

추천:

- DB에는 룰별/옵션별 행을 저장한다.
- API는 두 view를 제공한다.

```text
1. product-level: 제품별 통합 실제 성과
2. cafe24-rule-level: 파이썬 xlsx와 유사한 카페24 출력 행별 성과
```

초기 구현은 product-level을 우선하고, 필요하면 xlsx export에서 rule-level을 추가한다.

## 20. 구현 시 주의사항

1. 기존 Meta 업로드/소재 성과 기능을 깨지 말 것.
2. 카페24 주문 CSV는 개인정보 컬럼을 포함하므로 raw 저장 정책을 신중히 정할 것.
3. `+` 포함 옵션은 1+1로 처리한다. 이 요구사항은 확정이다.
4. 판매가가 38,900이어도 옵션이 `블랙+블랙`, `블랙+그레이` 등 `+` 조합이면 1+1이다.
5. 판매수는 `총 결제금액 > 0` 기준이 아니라 카페24 CSV의 `수량` 합계 기준이다.
6. `선불금`, `쿠폰`, `적립금` 등 결제수단 때문에 총 결제금액이 0이거나 낮아도 판매수에는 포함한다.
7. Meta 광고비는 기존 업로드된 current metric 기준으로 집계한다.
8. productId가 없는 광고비는 제품별 실제 성과에 포함하지 말고 별도 경고로 보여준다.
9. ProductCostRule의 적용일자 기준으로 원가/판매가를 조회한다.
10. 1+1/옵션별 가격 차이는 Cafe24ProductRule override 또는 ProductVariant 방식으로 처리한다.
11. 파이썬 기존 결과와 맞추려면 rule-level view도 제공한다.
12. 웹앱의 ROAS 단위는 기존 API와 맞춰 ratio인지 percent인지 통일한다.
