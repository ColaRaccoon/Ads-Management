## 목표

쿠팡 상품 설정 탭(`/coupang/products`)에서 이미 등록된 상품 설정을 목록에서 클릭해 수정할 수 있게 만든다.

## 권장 구현 방향

권장 방향은 기존 `PATCH /coupang/product-settings/:id`를 무리하게 확장하지 않고, "상품 설정 화면용 전체 저장 API"를 하나 추가하는 것이다.

이유:

- 기존 `updateProductSetting`은 의도적으로 매칭 규칙을 건드리지 않는다는 테스트가 있다.
- 프론트에서 상품 수정, 비용 규칙 추가, 매칭 규칙 수정을 여러 API로 순차 호출하면 중간 실패 시 일부만 저장될 수 있다.
- 사용자는 상품 설정 화면에서 한 번 저장했다고 느끼므로, 백엔드에서도 하나의 트랜잭션으로 처리하는 편이 안전하다.

권장 API:

- `PATCH /coupang/product-settings/:id/configuration`

선택적으로 신규 생성까지 같은 UX로 통합하려면 다음 API도 추가할 수 있다.

- `POST /coupang/product-settings/configuration`

하지만 이번 요청의 핵심은 "이미 설정된 항목 클릭 후 수정"이므로, 우선 `PATCH`만 추가해도 충분하다. 신규 생성 폼은 기존 `POST /coupang/product-settings`를 유지하고, 새 폼에 매칭 키워드까지 넣는다면 생성 성공 후 `POST /coupang/mapping-rules`를 호출하거나 별도 `POST /configuration`을 추가한다.

### 2. 서비스에 `updateProductConfiguration` 추가

파일: `apps/api/src/coupang/coupang.service.ts`

권장 구조:

```ts
async updateProductConfiguration(id: string, body: Record<string, unknown>) {
  return this.prisma.$transaction(async (tx) => {
    const product = await tx.coupangProduct.findUnique({ where: { id } });
    if (!product) {
      throw new NotFoundException({ code: "COUPANG_PRODUCT_NOT_FOUND", message: "Coupang product not found." });
    }

    const productData = await this.buildCoupangProductSettingUpdateData(tx, body);
    if (Object.keys(productData).length > 0) {
      await tx.coupangProduct.update({ where: { id }, data: productData });
    }

    const costRule = maybeCostRuleCreate(body);
    if (costRule?.create) {
      await tx.coupangCostRule.create({
        data: { ...costRule.create, coupangProductId: id }
      });
    }

    await this.upsertPrimaryCoupangProductRule(tx, id, body);

    const updated = await tx.coupangProduct.findUnique({
      where: { id },
      include: {
        group: true,
        productRules: { orderBy: [{ priority: "asc" }, { createdAt: "asc" }] },
        costRules: { orderBy: [{ effectiveFrom: "desc" }, { createdAt: "desc" }] }
      }
    });

    return updated ? { ...updated, costRules: sortCoupangCostRulesForSettings(updated.costRules) } : updated;
  });
}
```

실제 구현 시 타입 문제 때문에 `tx` 타입을 명시하거나 기존 스타일에 맞춰 `typeof this.prisma` 계열을 피하고 필요한 부분만 인라인으로 작성해도 된다.

`updateProductSetting` 안에 있는 product field 처리 로직을 그대로 복사하지 말고, 가능하면 private helper로 빼는 것이 좋다.

처리 필드:

`groupId` 처리 규칙은 기존과 동일해야 한다.

### 4. 매칭 규칙 upsert 정책

상품 설정 화면에서는 "대표 매칭 규칙 1개"만 수정하는 것으로 설계한다. 여러 개의 세부 규칙은 기존 `쿠팡 매핑관리` 탭에서 계속 관리한다.

대표 규칙 선택 기준:

1. 프론트가 `mappingRuleId`를 보내면 그 규칙을 수정한다.
2. `mappingRuleId`가 없으면 해당 상품의 활성 규칙 중 `priority asc`, `createdAt asc` 첫 번째 규칙을 대표 규칙으로 본다.
3. 활성 규칙이 없으면 해당 상품의 첫 번째 규칙을 사용할지, 새로 만들지 결정해야 한다. 권장은 "활성 규칙이 없고 includeKeywords가 있으면 새로 생성"이다.

검증:

- `mappingRuleId`가 넘어왔는데 그 규칙의 `coupangProductId`가 현재 상품 id와 다르면 `BadRequestException`을 던진다.
- `includeKeywords`가 넘어왔고 비어 있으면 기존 규칙을 빈 키워드로 업데이트하지 않는다. UI에서도 저장 버튼을 막아야 한다.
- 제외 키워드는 빈 배열 가능.
- 우선순위는 기존 `numberOrDefault(body.priority, 100)` 재사용 가능.

권장 helper:

```ts
private async upsertPrimaryCoupangProductRule(
  tx: Prisma.TransactionClient,
  productId: string,
  body: Record<string, unknown>
) {
  const hasRuleFields =
    body.mappingRuleId !== undefined ||
    body.includeKeywords !== undefined ||
    body.excludeKeywords !== undefined ||
    body.priority !== undefined;

  if (!hasRuleFields) return;

  const includeKeywords =
    body.includeKeywords !== undefined ? requiredStringArray(body.includeKeywords, "includeKeywords") : undefined;

  const ruleId = optionalString(body.mappingRuleId);
  const existing = ruleId
    ? await tx.coupangProductRule.findUnique({ where: { id: ruleId } })
    : await tx.coupangProductRule.findFirst({
        where: { coupangProductId: productId, isActive: true },
        orderBy: [{ priority: "asc" }, { createdAt: "asc" }]
      });

  if (existing && existing.coupangProductId !== productId) {
    throw new BadRequestException({ code: "COUPANG_RULE_PRODUCT_MISMATCH", message: "Mapping rule belongs to another product." });
  }

  if (existing) {
    await tx.coupangProductRule.update({
      where: { id: existing.id },
      data: {
        displayName: body.displayName !== undefined ? requiredString(body.displayName, "displayName") : undefined,
        includeKeywords,
        excludeKeywords: body.excludeKeywords !== undefined ? stringArray(body.excludeKeywords) ?? [] : undefined,
        priority: body.priority !== undefined ? numberOrDefault(body.priority, 100) : undefined,
        isActive: true
      }
    });
    return;
  }

  if (!includeKeywords || includeKeywords.length === 0) return;

  await tx.coupangProductRule.create({
    data: {
      coupangProductId: productId,
      displayName: optionalString(body.displayName) ?? "매칭 규칙",
      includeKeywords,
      excludeKeywords: stringArray(body.excludeKeywords) ?? [],
      priority: numberOrDefault(body.priority, 100),
      adEnabled: true,
      isActive: true
    }
  });
}
```

주의: 위 코드는 계획용 예시다. 실제 타입은 Prisma 버전에 맞춰 조정한다.

### 5. 비용 규칙 생성 정책

백엔드는 기존 `maybeCostRuleCreate(body)`를 재사용한다. 이 helper는 비용 필드가 하나라도 있으면 새 `CoupangCostRule`을 만든다.

중요한 주의점:

- 비용 규칙을 새로 만들 때 일부 필드만 보내면 나머지는 DB 기본값 0이 될 수 있다.
- 따라서 프론트는 비용 규칙을 생성할 때 비용 필드를 전부 보내야 한다.
- 상품명이나 매칭 키워드만 수정하는 저장에서는 비용 필드를 보내면 안 된다.

비용 필드 전체 목록:

- `salePriceKrw`
- `supplyPriceKrw`
- `productCostKrw`
- `salesFeeRate`
- `salesFeeKrw`
- `sellerShippingFeeKrw`
- `growthInboundFeeKrw`
- `growthShippingFeeKrw`
- `returnRate`
- `returnCostPerUnitKrw`
- `extraCostKrw`
- `effectiveFrom`
- `note`

사용자 요청에는 판매가/원가/수수료가 핵심으로 언급되었지만, 기존 계산이 배송비/반품비/기타비용까지 사용하므로 편집 폼에는 기존 비용 필드를 유지하는 것이 안전하다. 최소한 기존 값이 새 비용 규칙에서 0으로 날아가지 않게 전체 비용 스냅샷을 보내야 한다.

기준일 기본값:

- 기존 행을 편집 폼에 불러올 때 비용 값은 `costRules[0]`에서 채운다.
- `effectiveFrom`은 기본적으로 오늘 날짜로 넣는 것을 권장한다.
- 최신 비용 규칙의 `effectiveFrom`을 그대로 넣으면 사용자가 의도하지 않게 같은 기준일로 새 규칙을 추가할 수 있다.
- 단, 사용자가 과거 기준 보정을 원할 수 있으므로 날짜 입력은 수정 가능해야 한다.

### 6. 테스트 추가

파일: `apps/api/src/coupang/coupang.service.spec.ts`

기존 `describe("CoupangService product settings and mapping rules", ...)`에 추가하는 것이 자연스럽다.

추가할 테스트:

1. `updateProductConfiguration`이 상품명과 그룹을 수정하고 새 비용 규칙을 생성한다.
   - `coupangProduct.update` 호출 확인
   - `coupangCostRule.create` 호출 확인
   - `effectiveFrom`이 요청 날짜로 들어가는지 확인

2. 비용 필드가 요청에 없으면 새 비용 규칙을 만들지 않는다.
   - 상품명/매칭 키워드만 수정하는 경우 `coupangCostRule.create` 미호출

3. 기존 대표 매칭 규칙이 있으면 키워드/제외 키워드/우선순위를 update한다.

5. `mappingRuleId`가 다른 상품 소속이면 실패한다.

6. 빈 includeKeywords로 기존 규칙을 업데이트하려 하면 실패하거나 프론트에서 막는 정책을 테스트한다. 백엔드에서도 `requiredStringArray`로 막는 편이 안전하다.

`costDirty`는 비용 입력 변경 시 true로 바꾼다. 상품명/그룹/키워드만 바꿀 때는 false여야 하며, 이때 payload에 비용 필드를 보내지 않는다.

### 2. 상품 목록 행 클릭 시 폼에 값 채우기

현재 `DataTable`에는 row click 옵션이 없다. 두 가지 방법 중 하나를 선택한다.

권장:

- `apps/web/src/components/data-table.tsx`에 optional `onRowClick`, `rowClassName`, `getRowKey` props를 추가한다.
- 기존 사용처는 props를 안 넘기면 변화가 없어야 한다.

예시:

```tsx
export function DataTable<T>({
  rows,
  columns,
  empty = "...",
  footer,
  onRowClick,
  rowClassName,
  getRowKey
}: {
  rows: T[];
  columns: Column<T>[];
  empty?: string;
  footer?: ReactNode;
  onRowClick?: (row: T) => void;
  rowClassName?: (row: T) => string | undefined;
  getRowKey?: (row: T, rowIndex: number) => string | number;
}) {
  ...
}
```

클릭 가능한 행은 키보드 접근도 처리하면 좋다.

```tsx
<tr
  key={getRowKey ? getRowKey(row, rowIndex) : rowIndex}
  className={rowClassName?.(row)}
  onClick={onRowClick ? () => onRowClick(row) : undefined}
  tabIndex={onRowClick ? 0 : undefined}
  onKeyDown={(event) => {
    if (!onRowClick) return;
    if (event.key === "Enter" || event.key === " ") {
      event.preventDefault();
      onRowClick(row);
    }
  }}
>
```

주의:

- 기존 설정 목록의 제품그룹 select는 row click과 충돌한다.
- 권장 UX는 설정 목록에서 inline select를 제거하고, 그룹명만 표시한 뒤 행 클릭 또는 "수정" 버튼으로 폼에서 수정하게 하는 것이다.
- inline select를 유지한다면 select의 `onClick={(event) => event.stopPropagation()}`가 필요하다. 하지만 이번 요청은 "항목 클릭해서 수정"이므로 inline select 제거가 더 자연스럽다.

### 4. 저장 mutation 분기

현재 `save` mutation은 항상 `apiPost("/coupang/product-settings", ...)`만 호출한다. 편집 모드를 추가해야 한다.

권장:

```ts
const save = useMutation({
  mutationFn: () => {
    if (editingProductId) {
      return apiPatch(`/coupang/product-settings/${editingProductId}/configuration`, buildUpdatePayload(form, costDirty));
    }
    return apiPost("/coupang/product-settings", buildCreatePayload(form));

  onSuccess: () => {
```

`queryClient.invalidateQueries()`를 넓게 호출하는 이유:

- 상품명/그룹은 상품 설정, 대시보드, 일별 리포트, 이익 분석에 영향을 줄 수 있다.

- 매칭 키워드는 추후 rematch 전까지 기존 row에 바로 반영되지 않을 수 있지만, mapping rule 목록과 이슈 목록에는 영향을 준다.

### 5. payload 생성에서 비용 필드 전송 주의

편집 저장 payload:

```ts
function buildUpdatePayload(form: ProductForm, costDirty: boolean) {
  const payload: Record<string, unknown> = {
    displayName: form.displayName,
    standardName: form.displayName,
    groupId: form.groupId || null,
    mappingRuleId: form.mappingRuleId || undefined,
    includeKeywords: listValue(form.includeKeywords),
    excludeKeywords: listValue(form.excludeKeywords),
    priority: integerValue(form.priority) ?? 100
  };

  if (costDirty) {
    Object.assign(payload, {
      salePriceKrw: numberOrUndefined(form.salePriceKrw),
      supplyPriceKrw: numberOrUndefined(form.supplyPriceKrw),
      productCostKrw: numberOrUndefined(form.productCostKrw),
      salesFeeRate: numberOrUndefined(form.salesFeeRate),
      salesFeeKrw: numberOrUndefined(form.salesFeeKrw),
      sellerShippingFeeKrw: numberOrUndefined(form.sellerShippingFeeKrw),
      growthInboundFeeKrw: numberOrUndefined(form.growthInboundFeeKrw),
      growthShippingFeeKrw: numberOrUndefined(form.growthShippingFeeKrw),
      returnRate: numberOrUndefined(form.returnRate),
      returnCostPerUnitKrw: numberOrUndefined(form.returnCostPerUnitKrw),
      extraCostKrw: numberOrUndefined(form.extraCostKrw),
      effectiveFrom: form.effectiveFrom
    });
  }

  return payload;
}
```

핵심:

- `costDirty === false`이면 비용 필드를 절대 보내지 않는다.
- `costDirty === true`이면 비용 필드를 전부 보낸다.
- `effectiveFrom`은 비용 규칙을 새로 만들 때만 보낸다.

## 매칭 변경 후 rematch 정책

매칭 키워드/제외 키워드/우선순위를 수정해도 기존 업로드 row의 `coupangProductId`가 자동으로 즉시 바뀌지는 않는다. 기존 코드상 재매칭은 `POST /coupang/rematch`에서 수행한다.

선택지:

1. 저장 성공 후 자동 rematch까지 실행한다.
2. 저장 성공 후 "자동매핑 다시 계산" 버튼을 누르도록 둔다.

권장: 이번 작업에서는 자동 rematch를 하지 않는다.

이유:

- rematch는 기간 조건이 필요하다.
- 상품 설정 저장이 예상보다 느려질 수 있다.
- 기존 매핑관리 탭에 이미 rematch 버튼이 있다.

대신 저장 성공 후 UI에 짧게 "매칭 규칙 변경은 재계산 후 기존 데이터에 반영됩니다" 같은 메시지를 보여줄 수 있다. 단, 현재 디자인 지침상 과한 설명 문구는 피하고, warning strip 수준으로 짧게 처리한다.

## 수동 QA 체크리스트

- 설정 목록 행을 클릭하면 폼에 기존 값이 채워진다.
- 취소 버튼을 누르면 신규 생성 모드로 돌아간다.
- 상품명만 수정 저장하면 비용 규칙이 새로 생기지 않는다.
- 제품그룹만 수정 저장하면 비용 규칙이 새로 생기지 않는다.
- 매칭 키워드/제외 키워드/우선순위만 수정 저장하면 비용 규칙이 새로 생기지 않는다.
- 판매가/원가/수수료 등 비용 필드를 수정 저장하면 새 `CoupangCostRule`이 생성된다.
- 새 비용 규칙의 `effectiveFrom`이 폼의 기준일과 일치한다.
- 비용 수정 후 목록의 대표 비용이 새 규칙 값으로 보인다.
- 매칭 키워드가 쉼표 또는 줄바꿈 입력을 모두 배열로 저장한다.
- 제외 키워드는 비워도 저장된다.
- 매칭 키워드는 비어 있으면 저장이 막힌다. 기존 rule을 빈 키워드로 만들지 않는다.
- 제품그룹 select 조작이 행 클릭과 충돌하지 않는다.
- 모바일 폭에서도 폼/테이블 텍스트가 버튼 안에서 깨지지 않는다.

## 주의할 점

- 비용 규칙은 덮어쓰기 금지. 항상 새 기준일 규칙으로 추가한다.
- 비용 필드를 일부만 보내면 새 규칙의 나머지 비용이 0이 될 수 있다. 프론트에서 비용 수정 시 전체 비용 스냅샷을 보낸다.
- 상품명/그룹/키워드만 수정할 때 비용 필드를 보내면 불필요한 새 비용 규칙이 생긴다.
- 매칭 규칙은 상품별로 여러 개가 있을 수 있다. 상품 설정 화면에서는 대표 1개만 수정하고, 복잡한 다중 규칙 관리는 기존 매핑관리 탭 역할로 남긴다.
- `mappingRuleId`가 다른 상품의 규칙을 가리키면 반드시 거부한다.
- 기존 `updateProductSetting`이 매칭 규칙을 건드리지 않는다는 테스트는 유지하는 편이 좋다. 새 endpoint 테스트를 별도로 추가한다.
- `DataTable` 확장은 optional props로만 한다. 기존 페이지들이 모두 이 컴포넌트를 쓰므로 기본 렌더링이 달라지지 않아야 한다.
- 현재 일부 파일의 한글이 터미널에서 깨져 보일 수 있다. 파일 자체가 UTF-8이면 내용을 무리하게 갈아엎지 말고 필요한 부분만 수정한다.
