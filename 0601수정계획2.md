## 1. 이번 결정의 핵심 결론

현재 프로그램 요구사항은 `광고 세트 단위 CSV` 업로드를 중심으로 설계되어 있다. 이 구조는 광고세트의 타겟/예산/게재 상태를 보는 데는 유용하지만, 실제 광고 운영에서 가장 중요한 소재별 후킹, 카피, 썸네일, 상품 각도, 랜딩 전환 병목을 판단하기 어렵다.

앞으로의 정답 구조는 다음과 같다.

```text
광고 단위 CSV 1개 업로드
-> 광고 단위 원천 데이터 저장
-> 광고세트 단위 자동 집계
-> 캠페인 단위 자동 집계
-> 광고/광고세트/캠페인 각각의 액션 제안
```

즉, 매일 `광고세트 CSV`와 `광고 CSV`를 2개 올리는 구조로 가지 않는다. 매일 올리는 파일은 `광고 단위 리포트 CSV 1개`로 통일한다. 광고세트와 캠페인 분석은 이 광고 단위 데이터를 합산해서 만든다.

## 4. 현재 코드/문서에서 바꿔야 하는 방향

현재 `메타광고관리기_요구사항.html`에는 1차 목표가 `광고 세트 단위 CSV` 업로드로 되어 있다. 이 문서는 이제 낡은 설계다. 다음 세션에서는 요구사항 문서 또는 실제 앱 요구사항을 다음 방향으로 수정해야 한다.

기존 표현:

```text
광고 세트 단위 CSV 업로드
meta_adset_daily_metrics
date_start + adset_name unique
/api/uploads/meta-adset-csv
/api/metrics/adsets
```

수정 표현:

```text
광고 단위 CSV 1개 업로드
meta_ad_daily_metrics를 원천 데이터로 저장
adset/campaign 성과는 원천 광고 데이터를 집계해서 생성
date_start + campaign_id + adset_id + ad_id 또는 ad_name 조합으로 unique
/api/uploads/meta-ad-daily-csv
/api/metrics/ads
/api/metrics/adsets
/api/metrics/campaigns
```

현재 `make_agency_report.py`는 광고세트 중심으로 `by_adset` 집계를 만들고 있다. 이 로직은 버리지 말고, 입력 원천을 광고 단위로 바꾼 뒤 광고세트 집계는 파생 집계로 재작성한다.

현재 `make_meta_margin_daily_report.py`는 마진/손익 기준 운영 판단을 광고세트명 중심으로 보고 있다. 새 구조에서는 광고 단위 성과를 먼저 저장한 뒤, 제품 매칭과 마진 기준 판정은 다음 3단계에서 모두 가능해야 한다.

- 광고 단위: 소재/후킹/메시지 판단
- 광고세트 단위: 타겟/세팅/예산 판단
- 캠페인 단위: CBO/ASC 예산 배분 판단

## 5. 데이터 모델 수정 계획

PostgreSQL 기준 권장 모델은 아래와 같다. Prisma 또는 TypeORM 중 무엇을 쓰든 구조는 이 방향으로 잡는다.

### 5.1 upload_batches

CSV 업로드 이력을 저장한다.

필드:

- `id`
- `original_filename`
- `file_hash`
- `schema_version`
- `source_level`: 기본값 `ad`
- `report_start`
- `report_end`
- `row_count`
- `column_schema_json`
- `status`: `previewed`, `imported`, `failed`
- `uploaded_at`
- `imported_at`

중복 정책:

- 동일 `file_hash` 재업로드는 기본 차단
- 사용자가 원하면 `덮어쓰기`, `건너뛰기`, `새 버전 저장` 중 선택

### 5.2 meta_campaigns

캠페인 마스터.

필드:

- `id`
- `meta_campaign_id`: string, unique
- `campaign_name`
- `first_seen_at`
- `last_seen_at`

### 5.3 meta_adsets

광고세트 마스터.

필드:

- `id`
- `meta_adset_id`: string, unique
- `meta_campaign_id`
- `adset_name`
- `first_seen_at`
- `last_seen_at`

주의:

- 캠페인 ID와 광고세트 ID는 절대 number로 저장하지 않는다.
- 광고세트명이 바뀌어도 ID 기준으로 동일 광고세트로 추적한다.

### 5.4 meta_ads

광고 마스터.

필드:

- `id`
- `meta_ad_id`: string, nullable
- `meta_campaign_id`
- `meta_adset_id`
- `ad_name`
- `synthetic_ad_key`
- `first_seen_at`
- `last_seen_at`

현재 CSV에는 `광고 ID`가 없으므로 `synthetic_ad_key`를 만든다.

```text
synthetic_ad_key = sha1(meta_campaign_id + "|" + meta_adset_id + "|" + ad_name)
```

나중에 `광고 ID`가 들어오면 식별 우선순위는 다음과 같다.

```text
1순위: meta_ad_id
2순위: meta_campaign_id + meta_adset_id + ad_name
```

### 5.5 meta_ad_daily_metrics

광고 단위 일별 원천 성과 테이블. 이 테이블이 앞으로의 핵심 원천 데이터다.

필드:

- `id`
- `upload_batch_id`
- `date_start`
- `date_end`
- `meta_campaign_id`
- `campaign_name_snapshot`
- `meta_adset_id`
- `adset_name_snapshot`
- `meta_ad_id`
- `synthetic_ad_key`
- `ad_name_snapshot`
- `ad_delivery_status`
- `attribution_setting`
- `result_indicator`
- `result_count`
- `purchase_count`
- `spend_usd`
- `reach`
- `frequency`
- `impressions`
- `cpm_usd`
- `link_clicks`
- `shop_clicks`
- `cpc_link_usd`
- `ctr_link`
- `clicks_all`
- `ctr_all`
- `cpc_all_usd`
- `landing_page_views`
- `cost_per_lpv_usd`
- `quality_ranking`
- `engagement_rate_ranking`
- `conversion_rate_ranking`
- `raw_row_json`
- `created_at`

권장 unique:

```text
unique(date_start, meta_campaign_id, meta_adset_id, coalesce(meta_ad_id, synthetic_ad_key))
```

DB에서 `coalesce` unique 처리가 번거로우면 `ad_identity_key` 컬럼을 별도로 둔다.

```text
ad_identity_key = meta_ad_id if exists else synthetic_ad_key
unique(date_start, meta_campaign_id, meta_adset_id, ad_identity_key)
```

### 5.6 meta_adset_daily_metrics_view

별도 원천 테이블로 저장하지 말고, 우선은 view 또는 query aggregation으로 만든다.

집계 키:

```text
date_start + meta_campaign_id + meta_adset_id
```

집계 방식:

- `spend_usd`: 합계
- `purchase_count`: 합계
- `reach`: 원칙적으로 단순 합계는 중복 가능성이 있으므로 주의. 광고 단위 row의 reach를 더하면 같은 사람이 여러 광고에 노출된 중복이 생길 수 있다. Meta 광고세트 원본 reach와 완전히 같지 않을 수 있다.
- `impressions`: 합계 가능
- `link_clicks`: 합계 가능
- `clicks_all`: 합계 가능
- `landing_page_views`: 합계 가능
- `cpa`: `spend_usd / purchase_count`
- `ctr_link`: `link_clicks / impressions * 100`
- `cpc_link`: `spend_usd / link_clicks`
- `cpm`: `spend_usd / impressions * 1000`

주의: CTR, CPC, CPM은 CSV에 있는 값을 평균내면 안 된다. 반드시 분자/분모를 다시 합산해서 계산한다.

### 5.7 meta_campaign_daily_metrics_view

캠페인 단위도 광고 원천 데이터를 집계한다.

집계 키:

```text
date_start + meta_campaign_id
```

집계 방식은 광고세트 집계와 동일하다.

## 6. CSV 파서/검증 수정 계획

### 6.1 새 파서 이름

권장 클래스/파일명:

- `MetaAdDailyCsvParser`
- `MetaAdDailyCsvValidator`
- `MetaAdDailyImportService`
- `MetaMetricsAggregationService`

NestJS 기준 모듈:

- `UploadsModule`: 파일 업로드, 해시, preview, import
- `MetaAdsModule`: 캠페인/광고세트/광고 마스터 관리
- `MetricsModule`: 광고/광고세트/캠페인 성과 조회
- `MappingsModule`: 제품/단계 매칭
- `DecisionModule`: 자동 판정

### 6.2 파싱 규칙

- CSV는 UTF-8 또는 UTF-8 BOM을 허용한다.
- 컬럼명은 한글 원문 그대로 매핑한다.
- ID 컬럼은 문자열로 읽는다.
- 빈 값, `-`, 공백은 `null` 또는 0으로 정규화한다.
- 금액/횟수 지표는 숫자로 변환한다.
- 퍼센트 컬럼은 현재 샘플처럼 `3.56` 형태면 3.56%로 취급한다.
- 원본 행은 `raw_row_json`에 그대로 보관한다.

### 6.3 필수 컬럼 검증

업로드를 막아야 하는 필수 컬럼:

- `보고 시작`
- `보고 종료`
- `캠페인 이름`
- `캠페인 ID`
- `광고 세트 이름`
- `광고 세트 ID`
- `광고 이름`
- `광고 게재`
- `지출 금액 (USD)`
- `노출`

권장 경고 컬럼:

- `결과`
- `결과 표시 도구`
- `링크 클릭`
- `랜딩 페이지 조회`
- `CPC(링크 클릭당 비용) (USD)`
- `CTR(링크 클릭률)`

있으면 더 안전한 컬럼:

- `광고 ID`

`광고 ID`가 없으면 업로드를 막지는 않는다. 대신 preview 단계에서 다음 경고를 띄운다.

```text
광고 ID가 없어 campaign_id + adset_id + ad_name 조합으로 광고를 식별합니다.
같은 광고세트 안에 같은 광고명이 생기면 장기 추적이 부정확할 수 있습니다.
Meta 다운로드 설정에서 광고 ID 컬럼 추가를 권장합니다.
```

### 6.4 행 단위 검증

각 행마다 검증한다.

- `보고 시작`과 `보고 종료`가 날짜인지 확인
- `캠페인 ID`, `광고 세트 ID`가 비어 있지 않은지 확인
- `광고 이름`이 비어 있지 않은지 확인
- 지출, 노출, 클릭, 랜딩조회, 결과가 음수가 아닌지 확인
- 같은 파일 안에서 `date_start + campaign_id + adset_id + ad_identity_key` 중복이 있는지 확인
- 같은 ID가 서로 다른 이름으로 들어오면 경고

이번 샘플 검증 결과:

- `캠페인 ID -> 캠페인 이름` 매핑 문제 없음
- `광고 세트 ID -> 광고 세트 이름` 매핑 문제 없음
- `보고 시작 + 캠페인 ID + 광고 세트 ID + 광고 이름` 중복 없음
- 광고명만 기준으로 보면 `버닝웨이브바_02`, `버닝웨이브바_03`, `버닝슬라이드_07`은 여러 광고세트에 존재함

따라서 광고명 단독 key는 절대 쓰면 안 된다.

## 7. 분석 로직 수정 방향

### 7.1 광고 단위 분석

광고 단위는 소재 판단을 담당한다.

핵심 질문:

- 이 소재가 사람을 멈춰 세우는가?
- 클릭은 만들지만 구매 의도가 낮은가?
- 랜딩까지는 오는데 구매 설득이 약한가?
- 같은 광고명이 다른 광고세트에서 다르게 작동하는가?

기본 지표:

- spend
- purchase_count
- CPA
- impressions
- CPM
- link CTR
- link CPC
- landing page views
- landing view rate = `landing_page_views / link_clicks`
- purchase per LPV = `purchase_count / landing_page_views`
- purchase per click = `purchase_count / link_clicks`

자동 판정 예시:

- `노출 3,000 이상 + 링크 CTR 1% 미만`: 후킹 약함
- `링크 클릭 50 이상 + 랜딩조회율 60% 미만`: 클릭 품질 또는 랜딩 로딩 문제
- `랜딩조회 100 이상 + 구매 0`: 광고 메시지와 랜딩/오퍼 불일치 가능성
- `구매 3건 미만`: 승패 판단 보류
- `구매 10건 이상`: CPA 비교 가능
- `구매 20건 이상`: 승자 후보

### 7.2 광고세트 단위 분석

광고세트 단위는 타겟/세팅 판단을 담당한다.

핵심 질문:

- 어떤 타겟/세팅에서 소재가 잘 먹히는가?
- CBO 안에서 어떤 광고세트가 예산을 가져가는가?
- 같은 소재가 어떤 광고세트에서만 구매를 만드는가?
- 광고세트 내부에서 특정 광고만 예산을 독식하는가?

집계는 `meta_ad_daily_metrics`에서 만든다.

광고세트 화면에는 반드시 drill-down을 넣는다.

```text
광고세트 행 클릭
-> 해당 광고세트 안의 광고 목록 표시
-> 광고별 지출/구매/CPA/CTR/CPC/LPV 확인
```

### 7.3 캠페인 단위 분석

캠페인 단위는 예산 배분과 구조 판단을 담당한다.

핵심 질문:

- CBO/ASC 캠페인이 구매를 만들고 있는가?
- 캠페인 안에서 어떤 광고세트가 지출을 가져가는가?
- 캠페인 예산이 검증된 쪽으로 흐르는가, 테스트 쪽으로도 충분히 남는가?

이번 샘플 기준 캠페인 요약:

| 캠페인 | 광고세트 수 | 광고명 수 | 행 수 | 지출 | 구매 |
|---|---:|---:|---:|---:|---:|
| 버닝슬라이드_CBO | 10 | 16 | 17 | $68.72 | 5 |
| 버닝웨이브바_CBO | 3 | 4 | 8 | $39.70 | 8 |
| 플로우라이트_CBO_파트너십 | 1 | 7 | 7 | $13.52 | 0 |
| ASC_버닝슬라이드_CBO | 1 | 3 | 3 | $12.49 | 0 |
| ASC_플로우라이트_CBO | 1 | 3 | 3 | $8.03 | 0 |

## 8. 제품/단계 매칭 수정 계획

기존에는 광고세트명 기준으로 제품을 매칭하는 방향이었다. 새 구조에서는 매칭 우선순위를 둔다.

제품 매칭 우선순위:

```text
1. 광고 ID 또는 synthetic_ad_key 직접 매핑
2. 광고명 패턴 매핑
3. 광고세트 ID 직접 매핑
4. 광고세트명 패턴 매핑
5. 캠페인명 패턴 매핑
```

운영 단계 매칭 우선순위:

```text
1. 캠페인명에 ASC 포함 -> ASC
2. 캠페인명 또는 광고세트명에 CBO 포함 -> CBO
3. 수동 stage mapping 존재 -> 해당 stage
4. 미매칭 -> unknown
```

제품명 추정 예시:

- `버닝웨이브바` 포함 -> 버닝웨이브바
- `버닝슬라이드` 포함 -> 버닝슬라이드
- `플로우라이트` 포함 -> 플로우라이트
- `네이버GFA` 포함 -> 별도 테스트/외부채널 태그

매칭은 반드시 사용자가 수정할 수 있어야 한다. 자동 매칭이 틀리면 사용자가 한 번 고치고, 다음 업로드부터 같은 ID/패턴에 자동 적용되어야 한다.

## 9. API 수정 계획

기존 API 명칭이 광고세트 중심이면 아래처럼 바꾼다.

### 9.1 업로드

```text
POST /api/uploads/meta-ad-daily-csv
```

역할:

- CSV 파일 업로드
- 컬럼 검증
- 행 검증
- preview 생성
- import 실행 옵션 제공

하위 호환:

```text
POST /api/uploads/meta-adset-csv
```

이전 주소는 당장 제거하지 말고 deprecated 처리한다. 내부에서는 새 업로드 서비스로 연결하거나, 과거 광고세트 단위 파일을 별도 legacy parser로 처리한다.

### 9.2 미리보기

```text
GET /api/uploads/:id/preview
```

응답에 포함할 것:

- row_count
- column_count
- schema_version
- missing_required_columns
- warnings
- campaign_count
- adset_count
- ad_count
- duplicate_keys
- sample_rows

### 9.3 성과 조회

```text
GET /api/metrics/campaigns?from=&to=&productId=&stage=
GET /api/metrics/adsets?from=&to=&campaignId=&productId=&stage=
GET /api/metrics/ads?from=&to=&campaignId=&adsetId=&productId=&stage=
GET /api/metrics/ads/compare-by-name?adName=&from=&to=
```

`compare-by-name`은 같은 광고명이 여러 광고세트에서 어떻게 달리 작동하는지 보기 위한 API다. 예: `버닝웨이브바_02`가 3개 광고세트에서 각각 어떤 CPA/CTR을 보였는지 비교.

### 9.4 drill-down

```text
GET /api/metrics/adsets/:metaAdsetId/ads?from=&to=
GET /api/metrics/campaigns/:metaCampaignId/adsets?from=&to=
```

## 13. 주의할 함정

1. 광고명을 고유키로 쓰면 안 된다. 같은 광고명이 여러 광고세트에 들어갈 수 있다.
2. Meta ID를 number로 파싱하면 안 된다. 반드시 string으로 저장한다.
3. 광고 단위 reach를 더한 값을 광고세트 reach로 단정하면 안 된다. 중복 도달 가능성이 있다.
4. CTR, CPC, CPM을 단순 평균내면 안 된다. 합산 후 재계산한다.
5. 결과 컬럼이 항상 구매라고 가정하면 안 된다. `결과 표시 도구`가 구매 이벤트인지 확인한다.
6. 광고세트 파일과 광고 파일을 동시에 업로드하는 구조로 복잡하게 만들지 않는다. 원천은 광고 단위 1개 파일로 통일한다.
7. 과거 광고세트 단위 파일을 지원해야 한다면 legacy import로 분리하고, 새 표준 흐름과 섞지 않는다.

## 15. 최종 방향 한 줄 요약

앞으로 프로그램은 `광고세트 분석기`가 아니라 `광고 단위 원천 데이터를 기반으로 광고, 광고세트, 캠페인을 계층적으로 분석하는 Meta 광고 관리도구`가 되어야 한다. 매일 업로드할 파일은 광고 단위 CSV 1개이며, 광고세트와 캠페인 분석은 그 파일에서 자동 집계한다.
