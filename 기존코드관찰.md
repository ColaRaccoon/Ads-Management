## 1. 한눈에 보는 결론

이 프로젝트는 CSV 업로드 기반 Meta 광고 성과 분석/운영 판단 보조 웹앱이다. 광고 계정에 직접 ON/OFF나 예산 변경을 실행하지 않고, CSV 데이터를 저장/계산한 뒤 추천 판정과 사람이 남긴 변경 로그, 보고서 파일을 제공한다.

기술 스택은 npm workspaces 기반 TypeScript 모노레포다.

- `apps/api`: NestJS 11 + Prisma 6 + PostgreSQL API. CSV 업로드, 매칭, KPI/손익 계산, 자동 판정, 보고서 생성 담당.
- `apps/web`: Next.js 15 App Router + React 19 + React Query + Recharts 업무형 대시보드.
- `packages/shared`: 공통 문자열 유니언 타입만 있는 작은 패키지. 현재 앱 코드에서 적극적으로 사용되지는 않는다.

현재 폴더 자체에는 `.git` 디렉터리가 없다. `node_modules`, `apps/api/dist`, `apps/web/.next`, `apps/api/storage`가 이미 존재한다. 관찰할 때는 생성물인 `node_modules`, `dist`, `.next`, 런타임 저장소를 제외하고 읽는 것이 맞다.

## 2. 실행과 환경

루트 `package.json`은 workspaces를 사용한다.

```text
npm run dev              -> API와 Web을 병렬 실행
npm run dev:api          -> @meta-ads-performance/api start:dev
npm run dev:web          -> @meta-ads-performance/web dev
npm run lint             -> 각 workspace tsc --noEmit
npm run test             -> 각 workspace test
npm run build            -> 각 workspace build
npm run prisma:validate  -> scripts/prisma-validate.mjs
npm run prisma:generate
npm run prisma:migrate
npm run prisma:seed
```

포트와 기본 URL:

- API 기본 포트: `4000`
- Web dev 포트: `3100`
- Web 기본 API URL: `NEXT_PUBLIC_API_BASE_URL ?? "http://localhost:4000/api"`
- API 전역 prefix: `/api`

환경 파일:

- `.gitignore`는 `.env`, `.env.*`를 무시하고 `!.env.example`만 예외 처리한다.
- 실제로 `apps/api/.env`, `apps/web/.env` 파일이 존재한다. 민감값일 수 있으므로 이 문서에는 값을 기록하지 않았다.
- README는 `.env.example` 파일을 언급하지만 현재 파일 목록에는 보이지 않는다.

Prisma 검증 스크립트 `scripts/prisma-validate.mjs`는 `DATABASE_URL`이 없을 때 로컬 예시 URL을 주입한 뒤 `npx prisma validate --schema apps/api/prisma/schema.prisma`를 실행한다.

## 4. 백엔드 전체 구조

### 4.1 진입점

- CORS는 `http://localhost:<port>`와 `http://127.0.0.1:<port>` 허용.
- `PORT` env 없으면 `4000`.

`apps/api/src/app.module.ts`

- 실제 import 모듈: `CommonModule`, `UploadsModule`, `ProductsModule`, `MappingsModule`, `MetricsModule`, `DecisionsModule`, `ReportsModule`, `ChangeLogsModule`.
- `ConfigModule.forRoot({ isGlobal: true })` 사용.

### 4.3 도메인 계층

도메인 폴더는 외부 I/O 없이 계산과 파싱을 담당하는 순수 클래스/함수 중심이다. 이 프로젝트에서 가장 테스트하기 쉬운 영역이다.

`domain/adset-name-normalizer.ts`

- `normalizeName`: trim 후 연속 공백을 단일 공백으로 축소.
- `toKey`: normalize 후 lowercase.
- 광고세트명 매칭의 기준 key를 만든다.

`domain/date-number.ts`

- 날짜 문자열은 `YYYY-MM-DD`, `YYYY/MM/DD`, `YYYY. M. D.`를 지원한다.
- date-only는 `new Date("YYYY-MM-DDT00:00:00.000Z")`로 만든다.
- 숫자는 콤마, `%`, 공백, `$`, `₩`, `원`을 제거한 뒤 Number 변환.
- 퍼센트는 scale 변환 없이 `1.107011% -> 1.107011`로 보존한다.
- `safeDivide`는 비정상/0분모면 `null`.
- `dateRangeDays`는 양끝 포함 일수.

`domain/meta-csv.ts`

- Meta 광고세트 CSV 필수 컬럼 26개를 상수로 정의한다.
- 필수 컬럼:
  `보고 시작`, `보고 종료`, `광고 세트 이름`, `광고 세트 게재`, `기여 설정`, `결과`, `결과 표시 도구`, `도달`, `빈도`, `결과당 비용`, `광고 세트 예산`, `광고 세트 예산 유형`, `지출 금액 (USD)`, `종료`, `시작`, `노출`, `CPM(1,000회 노출당 비용) (USD)`, `링크 클릭`, `shop_clicks`, `CPC(링크 클릭당 비용) (USD)`, `CTR(링크 클릭률)`, `클릭(전체)`, `CTR(전체)`, `CPC(전체) (USD)`, `랜딩 페이지 조회`, `랜딩 페이지 조회당 비용 (USD)`.
- `CsvHeaderValidator`는 BOM 제거/trim 후 누락 컬럼을 찾는다.
- `MetaCsvParser.parseBuffer`는 `csv-parse/sync` 사용, `bom: true`, `columns: true`, `skip_empty_lines: true`.
- `parseRow`는 날짜, 광고세트명, 주요 숫자를 파싱하고 `ParsedMetaAdsetRow`를 만든다.
- `metricDate`는 현재 `dateStart`.
- `resultCount`는 CSV `결과` 컬럼을 그대로 구매수처럼 사용한다. seed에 `purchase_result_indicators` 설정이 있지만 현재 파서/계산에서 사용하지 않는다.
- `hashRecord`는 row JSON 문자열의 SHA-256.

`domain/matching.ts`

- `AdsetProductMatcher`: 수동 product history가 rule보다 우선한다. 그 다음 active rule을 priority 오름차순으로 평가한다.
- rule 타입:
  - `EXACT`: nameKey와 pattern 완전 일치.
  - `REGEX`: `new RegExp(pattern, "i")`.
  - 그 외 `CONTAINS`: includes.
- `AdsetStageMatcher`: 수동 stage history 우선. 없으면 광고세트명 token에서 `asc`, `cbo`, `sc`를 추론한다.
- stage token은 시작/끝 또는 공백/언더스코어/하이픈 경계가 있어야 한다. `classic` 안의 `sc`는 stage로 보지 않는다.

`domain/duplicate-policy.ts`

- `SKIP`: 기존 current metric이 있으면 새 metric import 안 함.
- `OVERWRITE`, `NEW_VERSION`: 둘 다 새 row를 current로 만들고 기존 current를 `isCurrent=false`로 바꾼다. 실제 차이는 note 문자열뿐이고, 기존 row를 물리적으로 덮어쓰지는 않는다.

`domain/margin-calculator.ts`

- `contributionBeforeAdsKrw = salePrice - vat - productCost - shipping - extraCost`.
- `breakEvenCpaKrw = contributionBeforeAdsKrw / adCostMultiplier`.
- target/watch/stop CPA는 break-even에 각 ratio를 곱한다.
- margin 계산:
  - `spendKrw = spendUsd * fxRateKrwPerUsd`
  - `revenueKrw = purchaseCount * salePriceKrw`
  - `grossCostKrw = purchaseCount * (vat + productCost + shipping + extraCost)`
  - `adCostWithMultiplierKrw = spendKrw * adCostMultiplier`
  - `marginKrw = revenueKrw - grossCostKrw - adCostWithMultiplierKrw`

`domain/period-metric-calculator.ts`

- 기간 KPI는 일별 ratio 평균이 아니라 기간 합계에서 재계산한다.
- CPA, CTR, CPC, ROAS는 합계 기반.
- `dataDays`는 metricDate distinct count.

`domain/decision-classifier.ts`

- 입력: stage, purchaseCount, spendKrw, cpaKrw, marginKrw, dataDays, CTR/LPV, CPA thresholds, good signal thresholds.
- 출력: 여러 decision을 동시에 반환할 수 있다.
- 전체/제품 scope에서는 margin 양수면 `PROFIT`, 음수면 `LOSS`.
- `SCALE`: 구매 있음, CPA가 target 이하, 마진 양수.
- `KEEP`: 구매 있음, CPA가 target 초과이지만 break-even 이하, 마진 0 이상.
- `WATCH`: 구매 있음, CPA가 break-even 초과 watch 이하, 마진 0 이상.
- SC에서 구매가 없어도 stop CPA 미만 지출이고 CTR/LPV 신호가 있으면 `WATCH`.
- `STOP_CANDIDATE`: 구매 없음+지출 stop 이상, 또는 구매 있음+CPA stop 초과, 또는 2일 이상 마진 음수.
- stage 이동 후보:
  - `SC_TO_CBO`: SC, 구매 1건 이상, CPA break-even 이하.
  - `SC_TO_ASC`: SC, 구매 2건 이상, CPA target 이하, 마진 양수.
  - `CBO_TO_ASC`: CBO, 2일 이상, 구매 2건 이상, CPA break-even 이하, 마진 양수.
  - `ASC_TO_SC`: ASC, 2일 이상, CPA stop 초과 또는 마진 음수.

`domain/domain.spec.ts`

- Vitest 테스트가 도메인 계층에 집중되어 있다.
- 날짜/숫자 파싱, CSV header/row 파싱, 마진 계산, 기간 KPI 재계산, 매칭 우선순위, 중복 정책, 판정 규칙을 검증한다.

### 4.4 업로드 흐름

컨트롤러: `uploads/uploads.controller.ts`

- `POST /api/uploads/meta-adset-csv`: multipart `file`, body `conflictPolicy`.
- `GET /api/uploads?take=50`
- `GET /api/uploads/:id/preview`
- `GET /api/uploads/:id/errors`

서비스: `uploads/uploads.service.ts`

CSV 업로드의 핵심 흐름:

1. file buffer 필수 검증.
2. conflictPolicy가 Prisma enum에 있는지 확인.
3. 파일 전체 SHA-256 계산.
4. 같은 file hash의 `UploadBatch`가 있으면 새 import를 하지 않고 기존 batch 요약을 반환한다. 이 경우 conflictPolicy와 무관하게 파일 단위 중복이 먼저 차단된다.
5. CSV headers/rows parse.
6. 원본 파일 저장:
   - `UPLOAD_STORAGE_DIR` env 또는 `./storage/uploads`
   - `process.cwd()` 기준으로 연/월 하위 폴더 생성
   - 파일명은 `hash앞12자리-safeOriginalName`
7. `UploadBatch` 생성, status `VALIDATING`.
8. header 필수 컬럼 검증 실패 시:
   - 누락 컬럼마다 `UploadRowError` 생성.
   - batch status `FAILED`.
   - `CSV_HEADER_INVALID` BadRequest.
9. row별 처리:
   - rowNumber는 header 다음 줄 기준이라 `index + 2`.
   - row parse issue가 있으면 validationStatus `ERROR`.
   - parse 성공 시 `MetaAdset` upsert.
   - `MappingsService.matchProduct`, `matchStage`로 product/stage 매칭.
   - `MetaAdset.currentProductId/currentStage/firstSeenOn/lastSeenOn` 업데이트.
   - product 미매칭이면 validationStatus `UNMATCHED`; 이 경우 warningCount와 validRowCount 모두 증가한다.
   - `UploadRow`와 필요 시 `UploadRowError` 저장.
   - parse 성공 row는 `importMetric`.
10. metric import:
    - 같은 `metricDate + metaAdsetId + isCurrent=true` current metric 조회.
    - DuplicatePolicyResolver 결과에 따라 skip 또는 새 version 생성.
    - supersede 시 기존 metric `isCurrent=false`, 새 metric 생성 후 기존 `supersededByMetricId`를 새 id로 업데이트.
11. batch 최종 status:
    - error 있고 imported 있으면 `PARTIAL`
    - error만 있으면 `FAILED`
    - error 없으면 `IMPORTED`

중요한 데이터 특성:

- `UploadRow`에는 rawRow, parsedRow, validationErrors를 모두 저장한다.
- `MetaAdsetDailyMetric`에는 원천 CSV 컬럼 대부분과 product/stage 매칭 결과, importVersion, isCurrent가 저장된다.
- `impressions`는 Prisma `BigInt` 필드다. raw Prisma 객체를 그대로 API로 내보내는 경로에서는 JSON 직렬화 주의가 필요하다.

### 4.7 KPI/대시보드 계산

컨트롤러:

- `GET /api/dashboard/summary?from&to&compare`
- `GET /api/dashboard/trends?from&to&groupBy=date|stage|product`
- `GET /api/metrics/adsets?from&to&productId&stage&decision`
- `GET /api/metrics/products?from&to`
- `GET /api/metrics/unmatched?from&to`

서비스: `metrics/metrics.service.ts`

핵심은 `decoratedMetrics -> decorate -> aggregate` 흐름이다.

`decoratedMetrics(fromDate, toDate)`

- current metric만 조회한다.
- product와 metaAdset relation include.
- 날짜/광고세트명 순으로 정렬.
- `decorate`로 비용/CPA rule과 마진 계산을 붙인다.

`decorate(metrics)`

- productId 목록으로 costRules/cpaRules를 한 번에 조회.
- metricDate에 유효한 rule은 `effectiveFrom <= date`이고 `effectiveTo`가 없거나 date 이상인 것 중 가장 최신 effectiveFrom.
- productId가 없으면 `ruleStatus=UNMATCHED`.
- cost rule이 없으면 `MISSING_COST_RULE` 또는 `MISSING_RULES`.
- cost rule이 있으면 `MarginCalculator.margin`으로 spendKrw, revenueKrw, marginKrw, CPA를 계산한다.
- cpa rule이 있으면 thresholds를 계산하고 `OK`, 없으면 `MISSING_CPA_RULE`.

`aggregate(rows)`

- `PeriodMetricCalculator`로 기간 합계 기반 KPI 계산.
- 하나라도 cost/cpa rule 누락이 있으면 totals에 `ruleStatus="CRITERIA_MISSING"`, 아니면 `OK`.

`dashboardSummary`

- 선택 기간 합계/평균/health/decisionSummary를 반환.
- 비교값은 다음 세 가지를 계산한다.
  - previousDay: 선택 기간 마지막 날의 전날 하루와 비교.
  - previousSamePeriod: 선택 기간 일수만큼 직전 기간과 비교.
  - firstDay: 선택 기간 마지막 날과 첫날 비교.
- `compare` query는 반환값에만 넣고 실제 비교 선택 분기에는 사용하지 않는다.

`dashboardTrends`

- groupBy가 `date`, `stage`, `product`일 수 있다.
- `stage/product`도 key를 `date:group`으로 만들어 일자별 group row를 반환한다.

`productMetrics`

- productId 있는 metric만 제품별 group.
- 제품별 totals, averages, thresholds, ruleStatus 반환.

`adsetMetrics`

- productId/stage query를 where에 반영.
- adset별 group 후 totals, thresholds, ruleStatus, firstSeenOn/lastSeenOn 반환.
- decision query가 있으면 해당 기간/decision의 `DecisionLog`를 찾아 metaAdsetId 기준으로 필터한다.
- 현재 반환 row의 `cpaDeltaVsPreviousDay`는 항상 null.

`unmatchedMetrics`

- productId null current metrics를 raw Prisma 객체로 반환한다.

### 4.8 자동 판정

## 5. Prisma 데이터 모델

Prisma schema는 `apps/api/prisma/schema.prisma`에 있고 DB는 PostgreSQL이다. 모든 enum은 `@@map`으로 snake_case enum 이름에 매핑된다. 대부분 모델도 `@@map`으로 snake_case table을 사용한다.

주요 enum:

- `UploadStatus`: `PENDING`, `VALIDATING`, `VALIDATED`, `IMPORTED`, `PARTIAL`, `FAILED`, `CANCELLED`
- `UploadLevel`: `ADSET`, `AD`, `CAMPAIGN`
- `ConflictPolicy`: `SKIP`, `OVERWRITE`, `NEW_VERSION`
- `RowValidationStatus`: `VALID`, `WARNING`, `ERROR`, `UNMATCHED`
- `MatchType`: `CONTAINS`, `EXACT`, `REGEX`, `MANUAL`
- `MatchSource`: `RULE`, `MANUAL`, `INFERRED`, `UNMATCHED`
- `AdStage`: `SC`, `CBO`, `ASC`, `UNKNOWN`
- `DecisionType`: `SCALE`, `KEEP`, `WATCH`, `STOP_CANDIDATE`, `SC_TO_CBO`, `CBO_TO_ASC`, `SC_TO_ASC`, `ASC_TO_SC`, `PROFIT`, `LOSS`
- `ReportType`: `DAILY_HTML`, `PERIOD_XLSX`, `CHANGE_LOG_XLSX`, `CPA_RULE_XLSX`

주요 모델 역할:

- `AppUser`: 현재 seed admin만 있고 auth 로직은 없다.
- `Product`: 제품 master. code unique, displayName, sortOrder, isActive.
- `ProductCostRule`: 제품 원가/판매가/환율/광고비 계수의 effective period rule.
- `ProductCpaRule`: target/watch/stop ratio의 effective period rule.
- `MetaAdset`: 광고세트 master. externalAdsetId는 현재 업로드 코드에서 사용하지 않고 null 기반 nameKey로 찾는다.
- `AdsetNameAlias`: schema에는 있으나 현재 서비스 로직에서 거의 사용되지 않는다.
- `ProductMatchRule`: 광고세트명으로 제품을 매칭하는 rule.
- `AdsetProductHistory`: 광고세트 수동 제품 이력.
- `AdsetStageHistory`: 광고세트 수동 stage 이력.
- `UploadBatch`: CSV 파일 단위 업로드 이력, hash, 기간, row/error count, status.
- `UploadRow`: CSV 원본 row와 parsed row, validation/matching 결과.
- `UploadRowError`: row/column 단위 검증 오류.
- `MetaAdsetDailyMetric`: current/versioned metric 저장소. 계산의 핵심 fact table.
- `DecisionRun`: 자동 판정 실행 단위.
- `DecisionLog`: 자동 판정 결과.
- `ChangeLog`: 사람이 수행한 운영 변경 기록.
- `ReportExport`: 보고서 파일 생성 이력.
- `AppSetting`: 시스템 설정 key/value JSON.

중요 인덱스/제약:

- `upload_batches.file_hash_sha256` unique. 파일 단위 중복 감지.
- `upload_rows(uploadBatchId,rowNumber)` unique.
- `meta_adset_daily_metrics(metricDate, metaAdsetId, importVersion)` unique.
- migration SQL에는 Prisma schema로 표현되지 않는 partial unique/index가 추가되어 있다.
  - externalAdsetId가 있을 때 `meta_adsets(platform, external_adset_id)` unique.
  - externalAdsetId가 null일 때 `meta_adsets(platform, adset_name_key)` unique.
  - current metric만 대상으로 `meta_adset_daily_metrics(metric_date, meta_adset_id)` unique.
  - current metric 조회용 date/product/stage/unmatched partial index.
  - unmatched upload row, nextCheckDate index.

Seed(`apps/api/prisma/seed.ts`)는 다음을 만든다.

- admin user.
- app settings:
  - timezone, default ratios, default conflict policy, good CTR/LPV 기준 등.
- 기본 제품 3개:
  - `burning-wavebar` / 버닝 웨이브바 / pattern `버닝웨이브바`
  - `burning-slide` / 버닝 슬라이드 / pattern `버닝슬라이드`
  - `flowlight` / 플로우라이트 / pattern `플로우라이트`
- 각 제품에 CONTAINS product match rule.

## 7. 코드 스타일과 설계 패턴

공통 스타일:

- TypeScript strict 사용.
- import는 double quote.
- 세미콜론 사용.
- 백엔드는 Nest controller/service/module 패턴.
- DTO class는 거의 없고 `Record<string, unknown>` body를 서비스 내부 helper로 검증한다.
- BadRequest/NotFoundException payload는 `{ code, message }` 형태.
- 날짜는 문자열 query를 date-only UTC Date로 바꿔 DB와 비교한다.
- Prisma Decimal/BigInt는 `numberFrom` 또는 `new Prisma.Decimal`로 명시 변환한다.
- 계산 로직은 `domain` 폴더에 클래스로 분리하고 서비스에서 Prisma I/O를 붙이는 구조가 선호된다.

백엔드에서 새 기능을 넣는 방식:

- 새 계산식: `src/domain/*`에 순수 계산 추가 -> `domain.spec.ts` 테스트 -> `MetricsService` 또는 `DecisionsService`에서 사용.
- 새 CSV 컬럼: `META_ADSET_REQUIRED_COLUMNS`, `ParsedMetaAdsetRow`, `MetaCsvParser.parseRow`, Prisma schema/migration, `UploadsService.importMetric`, UI table/report를 함께 수정해야 한다.
- 새 판정 규칙: `DecisionClassifier`에 추가하고 `domain.spec.ts`에 케이스를 추가한다.
- 새 API: Nest module/controller/service 패턴을 따른다. query date는 `parseDateRange`를 재사용한다.

프론트엔드 스타일:

- 거의 모든 route page가 `"use client"`다.
- Server Component 데이터 fetching보다 React Query client fetching을 사용한다.
- API 호출은 `lib/api.ts` helper를 통해 수행한다.
- UI 컴포넌트는 CSS class 기반이고, 일부 layout spacing은 inline style로 처리한다.
- table은 generic `DataTable`에 columns render 함수를 넘기는 방식.
- chart는 Recharts `ResponsiveContainer`를 사용한다.

## 9. 현재 구현상 주의점과 잠재 리스크

다음 항목들은 당장 “고쳐져 있다”가 아니라, 다음 작업자가 헷갈리지 않도록 기억해야 할 관찰 사항이다.

1. 이 폴더는 Git repo가 아니다. 변경 추적은 별도 방식이 필요하다.
2. `SettingsModule`은 존재하지만 `AppModule`에 import되지 않는다. 활성 `/api/settings`는 `ProductsModule`의 settings controller다.
3. `apps/web/src/components`에는 대소문자만 다른 중복 컴포넌트가 많다. 활성 app shell/page는 주로 소문자 파일을 쓴다.
4. 일부 구버전 컴포넌트는 CSS에 없는 `.btn`, `.badge.good` 같은 class를 사용한다.
5. `apps/web/src/types/api.ts`와 `apps/web/src/lib/api.ts`에 API 타입이 중복되어 있고, `types/api.ts`는 현재 활성 import가 거의 없다.
6. `packages/shared`의 타입은 현재 api/web에서 적극적으로 import하지 않는다.
7. `next.config.ts`와 `next.config.mjs`가 둘 다 있다. 내용은 동일하게 `reactStrictMode: true`.
8. README는 `.env.example`을 언급하지만 현재 파일 목록에는 `.env.example`이 보이지 않고 실제 `.env`가 있다.
9. `purchase_result_indicators` appSetting은 seed되지만 실제 구매수 계산은 CSV `결과` 숫자를 그대로 사용한다.
10. `adset_name_aliases` 테이블은 schema에 있지만 현재 주요 service 흐름에서는 사용되지 않는다.
11. `DuplicatePolicyResolver`에서 `OVERWRITE`와 `NEW_VERSION`은 둘 다 version 증가+기존 current supersede 방식이다. 물리적 overwrite가 아니다.
12. 파일 hash 중복은 conflictPolicy보다 먼저 적용된다. 동일 파일은 어떤 정책이어도 기존 batch 요약을 반환한다.
13. `unmatchedMetrics` 등 raw Prisma metric을 반환하는 API는 BigInt 필드 직렬화에 주의가 필요하다.
14. Adsets 화면의 product/stage select는 아직 API query와 연결되지 않았다.
15. Adsets 화면의 decision column은 아직 실제 decision을 표시하지 않는다.
16. Reports의 XLSX reportType들은 현재 같은 workbook 구조를 만든다.
17. 인증/권한 로직은 없다. `AppUser`는 schema/seed에 있지만 request user와 연결되어 있지 않다.
18. API `ValidationPipe`는 `whitelist:false`라서 body의 초과 필드를 제거하지 않는다.

## 11. 검증 명령

현재 설정상 의도된 검증 명령은 아래다.

```powershell
npm.cmd run prisma:validate
npm.cmd run test
npm.cmd run build
```

workspace별로는:

```powershell
npm.cmd --workspace @meta-ads-performance/api run test
npm.cmd --workspace @meta-ads-performance/api run lint
npm.cmd --workspace @meta-ads-performance/web run lint
```

주의:

- web `tsconfig.json`은 incremental tsbuildinfo를 `../../node_modules/.cache/web.tsbuildinfo`에 쓴다.
- API integration은 실제 `DATABASE_URL`/PostgreSQL 상태에 영향을 받을 수 있다.
- domain unit test는 DB 없이 실행되는 구조다.
