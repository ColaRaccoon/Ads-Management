# 0601 수정계획: Meta 광고 CSV 스냅샷 덮어쓰기와 active/inactive 필터

## 1. 배경

현재 프로그램은 매일 Meta 광고 세트 CSV를 업로드해서 DB에 저장한 뒤, 날짜 범위별로 광고세트 성과를 조회한다.

사용자가 확인한 문제는 두 가지다.

1. CSV에는 각 광고세트마다 `active`, `inactive` 값이 있지만, 화면/API에서 이 값을 기준으로 걸러보는 기능이 없다.
2. 특정 날짜 CSV에 없는 광고세트가 DB에는 남아 있으면, 같은 날짜를 조회할 때 계속 보일 수 있다.

이 문제를 해결하기 위해 기존의 `SKIP`, `OVERWRITE`, `NEW_VERSION` 업로드 정책을 운영자가 고르는 구조를 버리고, 업로드 CSV를 날짜별 "최신 전체 스냅샷"으로 간주하는 방향으로 변경한다.

핵심 방향:

```text
매일 업로드하는 CSV = 해당 날짜의 광고세트 현황 전체 스냅샷

새 CSV 업로드 시:
1. CSV에 들어온 row는 항상 최신 current metric으로 저장한다.
2. 같은 날짜에 DB에는 current로 남아 있지만 이번 CSV에는 없는 광고세트는 isCurrent=false로 숨긴다.
3. 화면/API는 기본적으로 active 광고세트만 보여준다.
4. 필요하면 all 또는 inactive 필터로 볼 수 있게 한다.
```

## 3. 최종 동작 정의

### 3.1 업로드 정책

앞으로 업로드 정책은 사용자가 고르지 않는다.

서버는 항상 다음처럼 동작해야 한다.

```text
effectiveConflictPolicy = OVERWRITE
```

단, 여기서 말하는 최종 의미는 기존 코드의 row 단위 `OVERWRITE`보다 더 넓다.

기존 `OVERWRITE`:

```text
같은 metricDate + metaAdsetId row가 CSV에 들어오면 그 row만 current 교체
CSV에 없는 기존 row는 그대로 current 유지
```

새 정책:

```text
CSV에 들어온 날짜별로 최신 스냅샷을 만든다.
CSV에 들어온 row는 최신 current로 저장한다.
CSV에 없는 기존 current row는 해당 날짜 조회에서 안 보이도록 isCurrent=false 처리한다.
```

### 3.2 active/inactive 필터

API와 화면에 다음 필터를 추가한다.

```text
deliveryStatus=active
deliveryStatus=inactive
deliveryStatus=all
```

기본값은 `active`를 추천한다.

이유:

- 운영자가 매일 보는 화면에서는 켜져 있는 광고세트만 보는 것이 자연스럽다.
- inactive row는 대개 지출/노출/구매가 0이라 운영 판단 화면을 지저분하게 만든다.
- 단, inactive 상태 확인이나 CSV 검증을 위해 `inactive`, `all` 조회도 가능해야 한다.

서버 기본값:

```text
deliveryStatus query가 없으면 active로 처리
```

주의:

- `active`, `inactive` 비교는 대소문자 차이를 흡수한다.
- 알 수 없는 값이 오면 `400 BadRequest`로 막는다.
- `all`이면 `deliveryStatus` 조건을 넣지 않는다.

### 3.3 날짜 필터와 CSV 누락 광고 처리

예시:

```text
기존 DB current:
2026-06-01 / A
2026-06-01 / B
2026-06-01 / C

새 CSV:
2026-06-01 / A
2026-06-01 / C

업로드 후 current:
2026-06-01 / A = true, 새 버전
2026-06-01 / B = false, 이번 CSV에 없어서 숨김
2026-06-01 / C = true, 새 버전
```

결과:

```text
/metrics/adsets?from=2026-06-01&to=2026-06-01
```

조회 시 B는 보이지 않는다.

## 4.3 importMetric은 항상 최신 버전 번호를 계산해야 한다

중요한 주의점:

현재 `MetaAdsetDailyMetric`에는 다음 unique가 있다.

```prisma
@@unique([metricDate, metaAdsetId, importVersion])
```

따라서 과거 version 1 row가 `isCurrent=false`로 남아 있어도, 새로 version 1을 다시 만들면 unique 충돌이 난다.

현재 코드:

```ts
const current = await this.prisma.metaAdsetDailyMetric.findFirst({
  where: {
    metricDate: input.parsedRow.metricDate,
    metaAdsetId: input.metaAdsetId,
    isCurrent: true
  },
  orderBy: { importVersion: "desc" }
});

let importVersion = 1;
if (current && duplicateDecision.supersedeExisting) {
  importVersion = current.importVersion + 1;
  await this.prisma.metaAdsetDailyMetric.update({ where: { id: current.id }, data: { isCurrent: false } });
}
```

문제:

- current가 없지만 과거 non-current version이 있으면 `importVersion=1`로 잡힐 수 있다.
- 이 경우 unique 충돌 가능성이 있다.

변경 방향:

```ts
const latest = await this.prisma.metaAdsetDailyMetric.findFirst({
  where: {
    metricDate: input.parsedRow.metricDate,
    metaAdsetId: input.metaAdsetId
  },
  orderBy: { importVersion: "desc" }
});

const current = latest?.isCurrent ? latest : await this.prisma.metaAdsetDailyMetric.findFirst({
  where: {
    metricDate: input.parsedRow.metricDate,
    metaAdsetId: input.metaAdsetId,
    isCurrent: true
  },
  orderBy: { importVersion: "desc" }
});

const importVersion = (latest?.importVersion ?? 0) + 1;
```

더 단순하게는 두 번 조회하지 말고 다음처럼 해도 된다.

```ts
const existingRows = await this.prisma.metaAdsetDailyMetric.findMany({
  where: {
    metricDate: input.parsedRow.metricDate,
    metaAdsetId: input.metaAdsetId
  },
  orderBy: { importVersion: "desc" },
  take: 5
});

const latest = existingRows[0] ?? null;
const current = existingRows.find((row) => row.isCurrent) ?? null;
const importVersion = (latest?.importVersion ?? 0) + 1;
```

그 다음 현재 row가 있으면:

```ts
if (current) {
  await this.prisma.metaAdsetDailyMetric.update({
    where: { id: current.id },
    data: { isCurrent: false }
  });
}
```

그리고 새 row를 항상 `isCurrent: true`로 만든다.

이제 `DuplicatePolicyResolver`는 업로드 경로에서는 사실상 필요 없다.

선택지:

1. 최소 변경: 파일과 테스트는 남겨두되 `uploads.service.ts`에서 사용하지 않는다.
2. 정리 변경: `duplicate-policy.ts`와 관련 테스트를 제거한다.

추천은 1번이다.

이유:

- 이번 작업의 목적은 운영 동작 수정이다.
- 불필요한 삭제로 변경 범위를 키우지 않는다.
- 기존 테스트를 크게 건드리지 않아도 된다.

## 4.4 이번 CSV에 없는 기존 current row 숨기기

수정 파일:

```text
apps/api/src/uploads/uploads.service.ts
```

새 helper를 추가한다.

추천 이름:

```ts
private async deactivateMissingSnapshotMetrics(input: {
  batchId: string;
  snapshotDates: Date[];
  includedKeys: Set<string>;
})
```

key 형식:

```ts
function snapshotMetricKey(metricDate: Date, metaAdsetId: string) {
  return `${formatDateOnly(metricDate)}:${metaAdsetId}`;
}
```

업로드 루프에서 CSV에 포함된 metric key를 모은다.

```ts
const snapshotDatesByKey = new Map<string, Date>();
const includedSnapshotKeys = new Set<string>();
```

`parsedRow && metaAdsetId`가 있고 row import가 성공했을 때:

```ts
const dateKey = formatDateOnly(parsedRow.metricDate);
snapshotDatesByKey.set(dateKey, parsedRow.metricDate);
includedSnapshotKeys.add(snapshotMetricKey(parsedRow.metricDate, metaAdsetId));
```

주의:

- parse error row는 포함하면 안 된다.
- `importMetric`이 실제로 imported=true를 반환한 row만 포함하는 것이 가장 안전하다.
- 앞으로는 항상 overwrite라 정상 row는 imported=true가 되어야 한다.

업로드 루프가 끝난 뒤, batch 상태를 업데이트하기 전에 또는 직후에 누락 row를 비활성화한다.

추천 조건:

```ts
if (errorCount === 0 && importedMetricCount > 0) {
  await this.deactivateMissingSnapshotMetrics({
    batchId: batch.id,
    snapshotDates: Array.from(snapshotDatesByKey.values()),
    includedKeys: includedSnapshotKeys
  });
}
```

왜 `errorCount === 0` 조건이 필요한가:

- CSV에 컬럼/숫자/날짜 오류가 섞인 상태에서 기존 current row를 숨기면, 잘못된 파일 하나로 그 날짜 데이터가 비어 보일 수 있다.
- 부분 성공 `PARTIAL` 업로드에서는 스냅샷 정리를 하지 않는 편이 안전하다.
- 사용자는 CSV 오류를 고친 뒤 다시 업로드하면 된다.

helper 구현 예시:

```ts
private async deactivateMissingSnapshotMetrics(input: {
  batchId: string;
  snapshotDates: Date[];
  includedKeys: Set<string>;
}) {
  if (input.snapshotDates.length === 0 || input.includedKeys.size === 0) {
    return 0;
  }

  const currentMetrics = await this.prisma.metaAdsetDailyMetric.findMany({
    where: {
      isCurrent: true,
      metricDate: { in: input.snapshotDates }
    },
    select: {
      id: true,
      metricDate: true,
      metaAdsetId: true,
      uploadBatchId: true
    }
  });

  const staleIds = currentMetrics
    .filter((metric) => !input.includedKeys.has(snapshotMetricKey(metric.metricDate, metric.metaAdsetId)))
    .map((metric) => metric.id);

  if (staleIds.length === 0) {
    return 0;
  }

  const result = await this.prisma.metaAdsetDailyMetric.updateMany({
    where: { id: { in: staleIds } },
    data: { isCurrent: false }
  });

  return result.count;
}
```

결과 response에 추가하면 좋다.

```ts
snapshotHiddenMetricCount
```

업로드 반환값 예시:

```ts
return {
  batchId: updated.id,
  status: updated.status,
  rowCount: updated.rowCount,
  validRowCount,
  warningCount,
  errorCount,
  importedMetricCount,
  skippedDuplicateCount: 0,
  snapshotHiddenMetricCount,
  unmatchedCount: warningCount,
  reportStart: reportStart ? formatDateOnly(reportStart) : null,
  reportEnd: reportEnd ? formatDateOnly(reportEnd) : null

```

주의:

- `skippedDuplicateCount`는 앞으로 항상 0에 가까워진다.
- UI나 리포트에서 아직 이 값을 쓰고 있다면 유지해도 된다.
- `snapshotHiddenMetricCount`는 운영자가 "이번 CSV에서 빠져 숨겨진 기존 row 수"를 확인하는 데 유용하다.

## 4.5 exact duplicate file hash 처리

현재 서비스는 파일 SHA-256이 완전히 같은 CSV를 다시 올리면 즉시 duplicate로 반환하고 재처리하지 않는다.

현재 코드:

```ts
const duplicated = await this.prisma.uploadBatch.findUnique({ where: { fileHashSha256 } });
if (duplicated) {
  return {
    duplicate: true,
    ...
  };
}
```

이 동작은 기본적으로 유지해도 된다.

이유:

- 같은 파일을 다시 올려도 결과는 같아야 한다.
- 불필요한 버전 증가를 막는다.

하지만 이번 수정 배포 직후, 과거에 같은 파일을 이미 잘못된 로직으로 업로드한 상태라면 같은 파일 재업로드가 막힐 수 있다.

그 경우 선택지는 두 가지다.

1. 운영 DB에서 해당 날짜를 새 파일명/내용으로 다시 업로드한다.
2. 별도 `force` 파라미터를 만들고 duplicate도 재처리하게 한다.

이번 작업에서는 `force`까지 만들 필요는 없다.

단, 문서화는 해두자.

## 4.6 active/inactive 필터 API 추가

수정 파일:

```text
apps/api/src/metrics/metrics.controller.ts
apps/api/src/metrics/dashboard.controller.ts
apps/api/src/metrics/metrics.service.ts
```

추천 타입:

```ts
type DeliveryStatusFilter = "active" | "inactive" | "all";
```

공통 parser/helper:

```ts
function parseDeliveryStatusFilter(value?: string): DeliveryStatusFilter {
  if (!value) return "active";
  const normalized = value.trim().toLowerCase();
  if (normalized === "active" || normalized === "inactive" || normalized === "all") {
    return normalized;
  }
  throw new BadRequestException({
    code: "INVALID_DELIVERY_STATUS",
    message: "deliveryStatus는 active, inactive, all 중 하나여야 합니다."
  });
}
```

`BadRequestException`을 쓰려면 import 변경:

```ts
import { BadRequestException, Injectable } from "@nestjs/common";
```

Prisma where helper:

```ts
function deliveryStatusWhere(filter: DeliveryStatusFilter): Prisma.MetaAdsetDailyMetricWhereInput {
  if (filter === "all") {
    return {};
  }
  return {
    deliveryStatus: { equals: filter, mode: "insensitive" }
  };
}
```

주의:

- Prisma + PostgreSQL에서 `mode: "insensitive"` 사용 가능하다.
- 타입 에러가 나면 다음처럼 lower-case 저장을 보장하는 방향도 가능하다.

```ts
deliveryStatus: input.parsedRow.deliveryStatus?.toLowerCase() ?? null,
```

그리고 조회는:

```ts
deliveryStatus: filter
```

단, 기존 DB에 `Active`, `ACTIVE` 같은 값이 섞였을 가능성을 고려하면 `mode: "insensitive"`가 더 안전하다.

`adsetMetrics` query 타입 변경:

```ts
async adsetMetrics(query: {
  from?: string;
  to?: string;
  productId?: string;
```

```ts
export function findMissingSnapshotMetricIds(
  currentMetrics: Array<{ id: string; metricDate: Date; metaAdsetId: string }>,
  includedKeys: Set<string>

    .filter((metric) => !includedKeys.has(snapshotMetricKey(metric.metricDate, metric.metaAdsetId)))
    .map((metric) => metric.id);
}
```

## 9. 주의사항

### 9.1 inactive row를 저장하지 않는 방식은 추천하지 않음

가능한 선택지:

```text
업로드 시 inactive row를 아예 저장하지 않는다.
```

이 방식은 추천하지 않는다.

이유:

- CSV 원본과 DB row 수가 달라져 검증이 어려워진다.
- inactive 상태 자체를 나중에 확인할 수 없다.
- "왜 Meta CSV에는 있는데 프로그램에는 없지?"라는 혼란이 생긴다.

추천:

```text
active/inactive 모두 저장한다.
조회 기본값만 active로 둔다.
```

### 9.2 MetaAdset.isActive와 혼동 금지

`MetaAdset.isActive`는 광고세트 마스터를 프로그램 내부에서 숨기거나 관리하기 위한 플래그에 가깝다.

이번 CSV의 `active/inactive`는 날짜별 광고 세트 게재 상태다.

따라서 필터 기준은 반드시:

```text
MetaAdsetDailyMetric.deliveryStatus
```

여야 한다.

### 9.3 부분 실패 업로드에서 snapshot cleanup 금지

CSV가 부분적으로 잘못되었는데 기존 current row를 숨기면 운영 화면이 잘못 비어 보일 수 있다.

따라서:

```ts
if (errorCount === 0 && importedMetricCount > 0) {
  await deactivateMissingSnapshotMetrics(...);
}
```

이 조건을 지킨다.

### 9.4 트랜잭션 범위

이상적으로는 다음을 하나의 transaction으로 묶는 것이 좋다.

```text
1. 기존 current row supersede
2. 새 metric create
3. 누락 current row deactivate
4. batch status update
```

다만 현재 업로드 서비스는 row별로 많은 DB 작업과 매칭 작업을 수행한다.

이번 수정에서는 큰 구조 개편보다 다음을 우선한다.

- 기존 흐름 유지
- 새 row import 완료 후 마지막에 누락 row만 deactivate
- 부분 실패 시 deactivate 생략

추후 리팩터링 때 transaction 범위를 정리한다.

### 9.5 같은 날짜 다중 batch

새 정책에서는 마지막으로 업로드된 정상 CSV가 그 날짜의 current snapshot이 된다.

따라서 운영 규칙은 다음과 같다.

```text
같은 날짜 CSV를 다시 올리면 이전 결과를 덮어쓴다.
마지막 정상 업로드가 진실이다.
```

이 규칙이 사용자 기대와 맞다.

### 9.6 과거 리포트와 결정 로그

`DecisionLog`, `ReportExport` 등은 당시 계산 snapshot을 JSON으로 저장하는 구조가 있다.

이번 작업은 `meta_adset_daily_metrics.isCurrent` 기준 조회를 바꾸는 것이다.

주의:

- 과거에 생성한 decision log/report는 자동으로 재계산되지 않는다.
- 새 CSV 업로드 후 필요한 경우 decision run/report를 다시 생성해야 한다.

## 11. 다음 세션 작업자에게

이번 작업의 핵심은 단순히 기존 conflict policy 기본값을 `OVERWRITE`로 바꾸는 것이 아니다.

반드시 다음 두 개를 함께 구현해야 한다.

```text
1. 들어온 row는 항상 최신 current version으로 저장
2. 같은 날짜인데 이번 CSV에 없는 기존 current row는 isCurrent=false 처리
```

그리고 active/inactive 필터는 `MetaAdset.isActive`가 아니라 `MetaAdsetDailyMetric.deliveryStatus` 기준이다.

가장 위험한 실수:

```text
업로드 시작하자마자 해당 날짜 current row 전체를 false로 바꾸는 것
```

이렇게 하면 CSV 파싱/환율/DB 저장 중 실패했을 때 화면에서 해당 날짜 데이터가 사라질 수 있다.

안전한 흐름:

```text
1. CSV 파싱
2. 환율 확보
3. row별 metric overwrite import
4. errorCount === 0일 때만 누락 current row 숨김
5. batch 완료 처리
```

이 흐름을 지키면 사용자가 원하는 "매일 CSV 기준으로 최신 상태만 보기"가 안정적으로 구현된다.
