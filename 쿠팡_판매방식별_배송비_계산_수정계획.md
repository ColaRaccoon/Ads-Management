## 3. 확정 요구사항과 해석

### 3.1 판매자 배송비 설정

기본 해석은 **쿠팡 상품별·원가 적용기간별 개당 설정값**이다. 전역 공통 설정이 아니다.

계산식:

```text
판매자배송비 = 판매자배송 판매수량 × 판매자 배송비 설정(개당)
```

DB에는 이미 `CoupangCostRule.sellerShippingFeeKrw` / `seller_shipping_fee_krw`가 존재하고, 상품 설정 API와 화면도 해당 필드를 전달하고 있다. 따라서 새 컬럼이나 Prisma 마이그레이션을 추가하지 않고 기존 필드의 의미와 UI를 명확히 한다.

화면 표기:

- 입력 항목명: `판매자 배송비 설정(개당)`
- 도움말: `판매자배송 수량 × 입력값으로 계산됩니다. 로켓그로스 판매에는 적용되지 않습니다.`
- 목록 헤더: `판매자 배송비 설정(개당)`
- 입력 조건: 0 이상의 유한한 원화 금액, 빈 값은 신규 원가 규칙 생성 시 기존 정책을 따르되 음수는 API에서 거부한다.

### 3.2 로켓그로스 배송비

기존 계산식을 유지한다.

```text
로켓그로스 배송비 = 로켓그로스 판매수량 × (그로스 입출고비 + 그로스 배송비)
```

### 3.3 혼합 판매 계산

동일 상품·동일 날짜에 두 판매방식이 함께 있으면 원본 행을 판매방식 기준으로 분리한다.

```text
판매자 구간 = 판매자배송 매출·순매출·수량·주문수
그로스 구간 = 로켓그로스 매출·순매출·수량·주문수
```

각 구간에 동일 날짜에 유효한 상품 원가 규칙을 적용하고, 구간별 비용을 합산한다.

```text
상품원가 합계 = Σ(구간 판매수량 × 상품 원가)
판매수수료 합계 = Σ(구간 순매출 × 수수료율) 또는 Σ(구간 수량 × 개당 수수료)
배송비 합계
  = 판매자배송 수량 × 판매자 배송비 설정
  + 로켓그로스 수량 × (그로스 입출고비 + 그로스 배송비)
반품비 합계 = Σ(구간 수량 × 반품률 × 개당 반품비)
기타비용 합계 = Σ(구간 수량 × 개당 기타비용)
VAT 합계 = Σ(구간 순매출 ÷ 11)
```

상품·날짜 단위 광고비는 구간마다 반복 차감하지 않고 합산 결과에 **정확히 한 번만** 차감한다.

```text
최종 정상판매 순이익
  = 전체 정상 순매출
  - 상품원가 합계
  - 판매수수료 합계
  - 배송비 합계
  - 반품비 합계
  - 기타비용 합계
  - VAT 합계
  - 상품·날짜 광고비 1회
```

광고 전환매출과 전환수량도 현재처럼 상품·날짜 합계를 사용한다. 판매방식별 자료가 없으므로 임의 배분하지 않는다. 유기적 매출과 ROAS는 합산 순매출/광고비 기준으로 한 번 계산한다.

## 4. 판매방식 정규화 정책

원본 문자열을 그대로 비교하면 동의어 때문에 불필요한 구간이 생길 수 있으므로 도메인 헬퍼에서 계산용 유형을 정규화한다.

권장 타입:

```ts
type CoupangFulfillmentMethod = "SELLER" | "GROWTH";
```

분류 기준:

- `growth`, `rocket`, `로켓`, `그로스` 포함: `GROWTH`
- 그 외 현재 호환 범위: `SELLER`

현재 `isGrowthSaleMethod()`와 동일한 호환성을 유지하되, 분류 함수를 외부에서도 사용할 수 있는 순수 함수로 분리한다. 원본 표시값은 보존하고 계산 키만 정규화한다.

향후 알 수 없는 판매방식을 엄격히 차단하려면 별도 `UNKNOWN` 타입을 도입할 수 있지만, 이번 작업에서 기존의 “그로스가 아니면 판매자배송” 동작을 갑자기 깨지 않는다.

## 5. 가구매와 혼합 판매 처리 정책

가구매가 없는 혼합 판매는 무조건 판매방식별로 정상 계산되어야 한다. 2026-07-20 충돌 10개 상품은 모두 이 경로에 해당한다.

가구매가 있는 경우에는 정상 판매에서 가구매 수량·매출을 어느 판매방식 구간에서 빼야 하는지가 필요하다.

처리 순서:

1. 가구매 스냅샷에 판매방식이 있고 해당 구간이 존재하면 그 구간에서 차감한다.
2. 원본 정상판매 구간이 하나뿐이면 그 구간에서 차감한다.
3. 원본이 혼합 판매이고 가구매 판매방식이 비어 있거나 일치하지 않으면 임의 비율 배분이나 첫 구간 선택을 하지 않는다.
4. 3번은 `MANUAL_PURCHASE_SALE_METHOD_REQUIRED_FOR_MIXED_SALES` 같은 구조화 경고로 해당 상품·날짜만 `INCOMPLETE` 처리한다.

가구매 원가 스냅샷에는 이미 `saleMethod`와 배송비가 저장되므로, 판매방식이 명확한 경우 기존 스냅샷 비용을 한 번만 차감한다.

이번 작업에서 가구매 데이터를 자동 수정하거나 기존 행의 판매방식을 추정하여 DB에 기록하지 않는다.

## 6. 권장 도메인 설계

### 6.1 판매 사실에 구간 정보 보존

`apps/api/src/domain/coupang-profit-pipeline.ts`

현재 `aggregateReportedSalesByProductDate()`가 합계와 `saleMethods`만 남기므로 판매방식별 매출·수량을 잃는다. 다음 중 한 구조로 바꾼다.

권장 구조:

```ts
type CoupangSalesSegment = {
  fulfillmentMethod: "SELLER" | "GROWTH";
  sourceSaleMethods: string[];
  salesKrw: number;
  cancelAmountKrw: number;
  netSalesKrw: number;
  salesQuantity: number;
  orderCount: number;
  lineCount: number;
};

type ReportedSalesFacts = {
  // 기존 상품·날짜 합계 필드 유지
  segments: CoupangSalesSegment[];
};
```

집계 키는 계속 `productId + date`를 사용하고, 그 안에서 `SELLER`와 `GROWTH` 구간을 각각 누적한다. API의 원본 합계 필드는 기존과 동일해야 한다.

### 6.2 가구매 조정 결과에도 구간 보존

`ActualSalesFacts`에 조정된 `segments`를 추가한다.

- 가구매 없음: reported segments 복사
- 단일 구간 가구매: 해당 구간에서 매출·순매출·수량 차감
- 혼합 구간 가구매: 5절의 귀속 정책 적용
- 음수 및 초과 검증은 현재 경고 정책 유지

합계 필드는 조정된 구간 합계와 반드시 일치해야 한다. 합계와 구간을 별도로 수정하여 불일치가 생기지 않도록 `sumSegments()` 같은 단일 헬퍼를 둔다.

### 6.3 판매방식별 순이익 계산 함수

`apps/api/src/domain/coupang-profit-calculator.ts`

기존 단일 구간 `calculateCoupangProfit()`은 유지하거나 내부 구성요소 함수로 사용한다. 판매방식별 구간을 합산하는 순수 함수를 추가하는 방식을 권장한다.

예시:

```ts
calculateCoupangProfitBySegments({
  segments,
  cost,
  ads,
  feeMode,
  includeReturnCost: true
})
```

구현 원칙:

- 각 구간의 상품원가·수수료·배송비·반품비·기타비·VAT를 계산
- 구간 계산에는 광고비 0원을 전달하거나 광고비를 계산기 외부에서 별도 추가
- 상품·날짜 광고비는 최종 합계에 한 번만 추가
- 유기적 매출, ROAS, `AD_CONVERSION_EXCEEDS_NET_SALES`는 합산 결과에서 한 번만 계산
- 돈 합계는 현재 정밀도 정책과 동일하게 유지하고 중간 단계에서 과도하게 반올림하지 않음

### 6.4 `NORMAL_SALE_METHOD_CONFLICT` 정책 변경

지원되는 `SELLER + GROWTH` 혼합은 더 이상 오류가 아니다.

- `calculateNormalCoupangProfit()`의 `reported.saleMethods.length > 1` 즉시 중단 로직 제거
- segments가 모두 유효하고 원가 규칙이 있으면 `COMPLETE`
- 혼합 여부는 표시용 `saleMethod: "MIXED"`로 유지 가능
- 원가 규칙 누락, 가구매 판매방식 귀속 불가 같은 실제 계산 불가 사유만 `INCOMPLETE`

## 7. API 서비스 및 응답 수정

`apps/api/src/coupang/coupang.service.ts`

### 7.1 `buildProductProfitRows()`

- 판매 원본을 상품·날짜별로 집계하면서 `SELLER`/`GROWTH` 구간 보존

### 7.2 감사 가능한 응답 필드

정확성 검증을 위해 다음 필드를 `ProductProfitRow`에 추가하는 것을 권장한다.

그룹/기간 집계 함수는 새 필드를 합산하되, 둘 중 하나가 계산 불가하면 기존 `strictSumNullable()` 정책을 따른다.

### 7.3 판매자 배송비 설정 API 검증

기존 `sellerShippingFeeKrw` 필드를 계속 사용한다.

- `PATCH /api/coupang/product-settings/:id/configuration`

검증:

## 10. 실제 데이터 완료 기준

2026-07-20 데이터를 기준으로 다음을 만족해야 한다.

1. 판매방식 충돌 10개 상품이 더 이상 `NORMAL_SALE_METHOD_CONFLICT`로 `INCOMPLETE`가 되지 않는다.
2. 각 상품의 판매자배송 수량에는 `sellerShippingFeeKrw × 수량`이 적용된다.
3. 각 상품의 로켓그로스 수량에는 `(growthInboundFeeKrw + growthShippingFeeKrw) × 수량`이 적용된다.
4. 광고비가 혼합 구간 수만큼 중복 차감되지 않는다.
5. 다른 오류가 새로 발생하지 않는다는 전제에서 `INCOMPLETE`는 11개에서 1개로 감소한다.
6. 제외 순매출은 6,397,210원에서 `한줄보호대 4개입` 152,400원만 남는다.
7. `한줄보호대 4개입`은 원가 규칙을 실제 값으로 등록하기 전까지 계속 `INCOMPLETE`여야 한다. 이번 코드 수정에서 임의 값으로 채우지 않는다.
8. 기존 정상 상품의 순이익 결과가 불필요하게 변하지 않는다. 판매방식이 하나뿐인 상품은 변경 전후 결과가 동일해야 한다.

## 12. 작업 시 주의사항

- 현재 저장소에는 2026-07-22 가구매 순이익 수정 관련 미커밋 변경이 많다. `git reset --hard`, `git checkout --`, 파일 전체 덮어쓰기를 하지 않는다.
- 기존 변경 위에 최소 범위로 이어서 구현한다.
- 원본 쿠팡 매출 행의 판매방식을 한 방식으로 강제 변경하지 않는다.

- 그로스 계산식은 변경하지 않는다.
